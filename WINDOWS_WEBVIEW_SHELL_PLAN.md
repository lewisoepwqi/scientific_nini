# Windows 壳层升级 v1 计划：内嵌 WebView2，保留 Python 后端与 React 前端

## 摘要

本轮只做壳层升级，不改业务前端架构、不改后端业务协议。

目标是把当前“后台服务 + 系统浏览器”升级为“后台服务 + 原生单窗口 + 系统托盘”，继续沿用现有 `PyInstaller + NSIS` 发布链路，支持 Windows 10/11，并由安装器自动补齐 WebView2 Runtime。

默认实现路线锁定为：

- 宿主技术：`pywebview + EdgeChromium(WebView2)`
- 后端形态：继续使用现有 FastAPI / WebSocket / 静态前端
- 打包形态：继续使用 `nini.exe + nini-cli.exe + NSIS`
- 默认体验：双击 `nini.exe` 直接打开内嵌窗口，不再默认拉起系统浏览器
- 范围边界：不重写 React UI；不引入 Tauri；不做自动更新；不做企业离线包；不做单实例治理

## 关键实现变更

### 1. 桌面宿主层

- 用 `pywebview` 替换当前 `windows_launcher` 中的 `webbrowser.open(...)` 主路径，宿主固定使用 EdgeChromium 后端。
- 保留现有托盘能力，托盘菜单调整为：`显示主窗口`、`在浏览器中打开`、`退出`。
- 主窗口行为固定为：
  - 启动后显示单窗口，窗口标题为 `Nini`
  - 关闭窗口时不退出进程，改为隐藏到托盘
  - 托盘点击可重新显示窗口
  - 只有点击托盘“退出”才真正结束后台服务
- 浏览器兼容入口保留，但降级为辅助路径，不再是默认启动路径。

### 2. 本地服务启动与地址策略

- 桌面模式下，launcher 启动本地后端时不再依赖固定 `8000` 端口，改为选择本机空闲端口并仅监听 `127.0.0.1`。
- 内嵌窗口始终加载 `http://127.0.0.1:{ephemeral_port}`。
- `nini-cli.exe start` 继续保留现有固定端口 / 可配置端口语义，作为调试与外部访问入口，不因桌面壳升级改变。
- 若后端在超时时间内未启动成功，launcher 直接弹原生错误框并退出，不进入空白窗口。

### 3. WebView2 Runtime 与安装器

- NSIS 安装器新增 WebView2 Runtime 检测步骤。
- 若系统缺失 Runtime，安装器静默调用微软 Evergreen Bootstrapper 自动安装；安装失败则给出明确阻断提示并终止安装。
- v1 不内嵌完整离线 Runtime；企业离线交付留到后续迭代。
- 运行期 launcher 也要做二次检测：如果 Runtime 缺失且用户绕过安装器直接运行 `nini.exe`，给出明确提示，不尝试回退到系统浏览器。

### 4. 前端与后端边界

- 不修改现有 HTTP API、WebSocket 事件模型、前端状态机语义。
- 不引入新的前后端协议层；WebView 直接加载现有静态前端，由前端继续通过 `window.location.host` 访问同源 API / WebSocket。
- 允许做极小的桌面端兼容调整，但仅限于：
  - 首屏加载态更稳
  - 文件选择 / 下载在 WebView2 下验证并修正
  - 窗口尺寸变化下的布局抖动修正
- 不做业务 UI 重构，不做桌面专属页面。

### 5. 打包与发布

- 继续保留双入口：
  - `nini.exe`：GUI 桌面入口
  - `nini-cli.exe`：CLI 调试 / 运维入口
- `nini.spec` 需要新增 WebView 宿主依赖收集，并保持现有静态资源、Chrome、模型缓存、Ollama bundle 逻辑不变。
- `build_windows.bat` 继续作为一键构建入口，新增 WebView 宿主依赖安装。
- `packaging/README.md` 更新为“桌面窗口”为默认形态，不再描述“默认打开系统浏览器”。

## 对外接口与行为变化

- 不变更任何现有后端 HTTP API、WebSocket 事件类型、数据库结构。
- `nini.exe` 的默认行为从“打开浏览器”改为“打开内嵌桌面窗口”。
- 保留外部浏览器兼容入口，建议新增一个显式启动参数用于调试：
  - `nini.exe --external-browser`
- CLI `nini start/init/doctor` 行为保持兼容，不因桌面壳升级改变。

## 测试与验收

### 核心验收场景

1. Windows 10/11 干净环境安装，安装器可检测并补齐 WebView2 Runtime。
2. 双击 `nini.exe` 后不弹系统浏览器，直接打开内嵌窗口并进入现有首页。
3. 首次启动、已有用户数据启动、已配置 Ollama 启动三种路径都可正常进入窗口。
4. WebSocket 对话、文件上传、图表展示、文件预览、下载导出在 WebView2 宿主下行为与浏览器模式一致。
5. 点击窗口关闭按钮后应用隐藏到托盘；从托盘可恢复窗口；点击“退出”后后台服务被真正清理。
6. 使用 `nini.exe --external-browser` 可回退到原浏览器模式，便于问题排查。
7. `nini-cli.exe doctor/start` 保持可用，不受桌面壳路径影响。

### 建议测试类型

- 后端：现有 `pytest -q` 不变，重点确保 API / WebSocket 契约不回归。
- 前端：现有 `npm run build` 不变；补充一组桌面壳冒烟清单，不依赖重写组件测试。
- 打包：增加 Windows 安装后手工验收清单，覆盖 Runtime 缺失、Runtime 已存在、升级安装三种场景。

## v2 / v3 后续迭代路线

### v2：桌面增强版

- 补单实例治理：二次启动时唤醒已有窗口而不是新开后台进程。
- 增加原生菜单与窗口状态记忆。
- 增加退出确认与后台运行提示。
- 增加原生日志查看入口与诊断导出。
- 评估自动更新机制，但此阶段仍不改业务 UI 技术栈。

### v3：企业与离线分发版

- 提供企业离线安装包，内置 WebView2 Runtime。
- 规划内网环境下的离线依赖交付：Chromium、字体、可选模型缓存、可选 Ollama bundle。
- 评估签名、策略分发、静默安装参数与管理员部署文档。
- 若未来确有必要，再评估把少量壳层能力改为更强桌面框架，但不触碰 React 主体。

## 假设与默认值

- 目标用户主要是 Windows 10/11 桌面用户。
- 可以接受安装器联网补齐 WebView2 Runtime。
- 当前 `FastAPI + React + WebSocket` 是保留资产，不在 v1 改造范围内。
- v1 以“替代系统浏览器”为唯一核心目标，不捆绑自动更新、单实例、企业离线部署等增强项。
- 实现阶段若 `pywebview` 在打包或托盘协同上出现阻断，再退回备选方案 `WinUI/WPF 壳 + 本地 URL 承载`；但当前默认不走这条线。
