# 可信输出与人工复核交互规范

本文档用于将 [nini-vision-charter.md](/home/lewis/coding/scientific_nini/docs/nini-vision-charter.md) 中的“输出等级、风险等级、证据可溯源、人工复核门”要求，落成一份面向前端交互与运行时联动的正式规范。

本文档回答三个问题：

1. 用户在界面中应该如何理解一条输出“能不能用、能用到什么程度”。
2. 哪些场景必须触发人工复核，复核前后如何交互。
3. 当前实现已具备什么，仍缺什么，后续该如何接入。

## 1. 文档目标

本规范的目标不是重新定义治理规则，而是把现有战略与模型层的规则变成用户可见、可操作、可审计的交互契约。

### 1.1 适用范围

适用于以下输出与交互对象：

- 聊天主回复
- Skill 步骤输出
- 报告、图表、文档等可导出产物
- 文献整理、实验设计、数据分析、论文写作四个重点阶段中的中高风险结果

### 1.2 非目标

以下内容不在本文档内：

- 不定义底层统计或文献方法是否正确
- 不替代高风险 Capability / Skill 的评审规范
- 不替代 Skill 契约层的字段设计
- 不单独定义最终导出中心的页面结构

## 3. 核心设计原则

### 3.1 输出等级必须回答“能用到什么程度”

输出等级不是装饰性徽章，而是用户判断结果边界的主要依据。

### 3.2 风险等级必须回答“错了会造成什么后果”

风险等级不等同于输出等级。一个内容可以是 `O2 草稿级`，同时也是 `high` 风险。

### 3.3 人工复核门必须是交互动作，不只是说明文字

只写“请人工复核”不算复核机制。用户必须在关键节点明确确认或取消。

### 3.4 证据状态必须影响结果呈现

有来源、有证据、不确定、离线草稿，这些状态必须改变用户看到的 UI，而不是只在内部逻辑存在。

### 3.5 默认从保守侧表达

若对风险级别、输出成熟度或证据充分性存在争议，默认按更保守的方式提示与阻断。

## 4. 概念模型

## 4.1 输出等级

| 等级 | 名称 | 用户可理解定义 | 前端默认语义 |
| --- | --- | --- | --- |
| `O1` | 建议级 | 方向性建议，仅供参考 | 不建议直接复制到正式材料 |
| `O2` | 草稿级 | 可编辑初稿，结构较完整 | 可以继续加工，但不能当作已验证结论 |
| `O3` | 可审阅级 | 方法与来源较完整，适合人工审阅 | 可以进入审阅流程，但仍需专业复核 |
| `O4` | 可导出级 | 达到导出标准的结构化产物 | 可以导出，但仍需人工终审，不代表最终正确 |

### 4.2 风险等级

| 等级 | 名称 | 用户可理解定义 | 典型示例 |
| --- | --- | --- | --- |
| `low` | 低风险 | 错误主要影响表达与效率 | 格式整理、通用写作润色 |
| `medium` | 中风险 | 错误会影响草稿质量或资料完整性 | 文献归纳、引用整理 |
| `high` | 高风险 | 错误会影响研究方法、统计判断或投稿策略 | 样本量估计、统计解释 |
| `critical` | 极高风险 | 错误可能影响患者安全、伦理合规或重大结论 | 临床建议、伦理判断、因果背书 |

### 4.3 证据状态

前端需要引入一个显式“证据状态”概念，至少支持以下四类：

| 状态 | 含义 | 默认视觉语义 |
| --- | --- | --- |
| `verified` | 有明确来源，结论与证据对应 | 正常展示 |
| `partial` | 有部分来源，但覆盖不完整 | 提醒审阅 |
| `missing` | 关键结论缺少来源或方法依据 | 黄色警示 |
| `offline_limited` | 受离线或降级影响，只能输出草稿级内容 | 明确降级提示 |

## 5. 界面对象与显示规则

### 5.1 聊天主回复

每条主回复至少需要支持以下元信息展示：

| 信息 | 是否必需 | 说明 |
| --- | --- | --- |
| 输出等级 | 是 | 所有主回复都应可标注 |
| 风险等级 | 条件必需 | `medium` 及以上建议显示，`high/critical` 必须显示 |
| 证据状态 | 条件必需 | 涉及文献、事实、统计结论时必须显示 |
| 人工复核提示 | 条件必需 | 命中复核条件时必须显示 |

#### 展示位置建议

1. 消息头部右侧显示等级徽章
2. 消息正文下方显示风险与证据摘要条
3. 需要复核时，在消息底部显示阻断式确认区域

### 5.2 Skill 进度面板

当前 [SkillProgressPanel.tsx](/home/lewis/coding/scientific_nini/web/src/components/SkillProgressPanel.tsx) 已支持步骤级 `review_required`，这是现有最接近规范的交互。

后续要求：

| 项目 | 要求 |
| --- | --- |
| 步骤级风险信息 | 在待确认步骤中显示风险摘要 |
| 待确认原因 | 不只显示“待确认”，还要说明为什么需要确认 |
| 证据状态 | 若因证据不足触发复核，应明确显示 |
| 决策结果 | 用户确认或取消后，界面应留下可追踪记录 |

### 5.3 导出前确认

以下对象在进入导出或“最终摘要/外发材料”前，应统一走导出前复核门：

- 报告导出
- 文档导出
- 论文草稿导出
- 最终摘要写回
- 对外发送的结构化产物

#### 导出前复核门必须包含

| 字段 | 是否必需 | 说明 |
| --- | --- | --- |
| 待导出内容摘要 | 是 | 用户正在导出的内容是什么 |
| 输出等级 | 是 | 当前内容成熟度 |
| 风险等级 | 是 | 内容风险级别 |
| 证据状态 | 是 | 来源与依据是否完整 |
| 风险说明 | 是 | 为什么需要复核 |
| 用户确认结果 | 是 | 确认导出 / 返回修改 |

## 6. 交互流程规范

### 6.1 低风险输出流程

适用：`low` 风险，且不命中强制复核场景。

流程：

1. 输出消息生成
2. 展示输出等级
3. 如有来源则展示证据状态
4. 允许用户继续使用、复制或导出

规则：

- 不弹出阻断式确认
- 仍需保留等级和来源信息

### 6.2 中风险输出流程

适用：`medium` 风险，或证据状态为 `partial`。

流程：

1. 输出消息生成
2. 展示输出等级 + 风险等级 + 证据状态
3. 显示轻量提示：“建议审阅后再用于正式材料”
4. 如进入导出动作，则触发导出前复核门

规则：

- 默认不阻断继续对话
- 阻断导出或写入最终摘要

### 6.3 高风险输出流程

适用：`high` 风险，或命中强制人工复核场景。

流程：

1. 输出消息生成
2. 展示输出等级 + 高风险标签
3. 紧随消息显示“需人工复核”区域
4. 若为 Skill 步骤，则进入 `review_required`
5. 未确认前，不允许进入最终摘要或导出

规则：

- 允许继续查看与讨论
- 不允许无确认直接导出为最终材料

### 6.4 极高风险输出流程

适用：`critical` 风险。

流程：

1. 输出内容仅允许到 `O1/O2` 或受信任上限控制
2. 消息顶部显示强警示
3. 明确声明“不能替代专业判断”
4. 默认禁止进入导出或最终摘要
5. 如需继续，只能以草稿形式保留，不得包装为最终建议

规则：

- 默认阻断导出
- 禁止使用弱提示替代强警示

## 7. 强制人工复核触发规则

以下情况必须触发复核门：

1. 输出涉及样本量计算与关键参数推荐
2. 输出涉及研究方案定稿
3. 输出涉及统计结论的最终解释
4. 输出涉及投稿回复与审稿意见处理
5. 输出涉及期刊适配与投稿建议
6. 输出涉及临床解释、伦理建议或因果判断
7. 输出即将进入最终摘要、导出文档或外部发送材料

### 7.1 复核门交互要求

复核门不能只有两个按钮，必须同时展示以下内容：

| 区域 | 内容 |
| --- | --- |
| 内容摘要 | 用户将继续或导出的内容是什么 |
| 风险说明 | 为什么该内容风险高 |
| 证据状态 | 有无来源、来源是否完整 |
| 建议动作 | 建议补充来源、复查参数或请专业人员确认 |
| 操作按钮 | `确认继续` / `返回修改` |

### 7.2 用户动作后的系统行为

| 用户动作 | 系统行为 |
| --- | --- |
| 确认继续 | 记录确认事件，允许进入后续步骤或导出 |
| 返回修改 | 中止进入最终链路，保留当前结果供用户修改 |
| 关闭或忽略 | 视为未确认，不得静默继续进入最终链路 |

## 8. 文案规范

### 8.1 必须避免的表述

- “最终成果，可直接用于报告或论文”
- “结论已验证”
- “推荐最终版本”
- “可以直接提交”

### 8.2 建议使用的表述

| 场景 | 建议文案 |
| --- | --- |
| `O2` 草稿级 | `这是可编辑草稿，建议在使用前补充和修正。` |
| `O3` 可审阅级 | `方法与来源较完整，建议人工审阅后使用。` |
| `O4` 可导出级 | `已达到导出标准，但仍需人工终审，不代表免复核。` |
| 高风险提示 | `该内容可能影响研究判断，需人工复核后再进入正式材料。` |
| 证据不足 | `当前结论的来源或依据不完整，请勿直接用于最终摘要。` |
| 离线降级 | `当前处于离线或降级模式，仅提供草稿级内容。` |

## 9. 视觉与组件建议

### 9.1 徽章层

至少需要三类徽章：

- 输出等级徽章
- 风险等级徽章
- 证据状态徽章

规则：

- 输出等级始终可见
- `high/critical` 风险必须使用高对比样式
- 证据缺失状态不能和普通提示样式混淆

### 9.2 卡片层

建议新增两个统一卡片：

1. `可信输出卡`
2. `人工复核卡`

#### 可信输出卡

展示：

- 输出等级
- 风险等级
- 证据状态
- 方法或来源摘要

#### 人工复核卡

展示：

- 风险说明
- 待确认内容摘要
- 证据状态
- `确认继续 / 返回修改`

### 10.2 建议新增或补齐的前端状态

## 11. 当前实现差距

### 11.1 已有部分

| 能力 | 当前实现 |
| --- | --- |
| 输出等级枚举和元数据 | 已在后端模型层存在 |
| 聊天消息输出等级显示 | 已在 `MessageBubble` 中存在 |
| 输出等级首次说明 | 已在 `OutputLevelExplainer` 中存在 |
| Skill 步骤待确认 | 已在 `SkillProgressPanel` 中存在 |

### 11.2 主要缺口

| 缺口 | 影响 |
| --- | --- |
| `O4` 文案仍偏乐观 | 容易被理解为可直接最终使用 |
| 主回复缺少风险等级显示 | 用户无法判断高风险内容 |
| 主回复缺少证据状态显示 | 用户无法区分“有依据”和“仅生成” |
| 缺少导出前复核门 | 高风险内容可能直接进入导出链路 |
| 复核原因不透明 | 当前步骤级待确认缺少原因说明与风险摘要 |

## 12. 研发拆解建议

### P0

1. 修正输出等级文案，统一与愿景文档对齐。
2. 为主回复增加风险等级与证据状态展示能力。
3. 为高风险主回复增加统一的“需人工复核”区域。
4. 为报告/文档/草稿导出增加导出前复核门。

### P1

1. 扩展 `SkillProgressPanel`，补充复核原因、证据状态和确认记录。
2. 补齐主回复与产物的 review 事件与状态。
3. 将证据状态与引用视图打通。

### P2

1. 引入统一的 `可信输出卡` 和 `人工复核卡` 组件。
2. 将离线/降级行为与可信度变化联动到 UI。

## 13. 验收标准

| 验收项 | 标准 |
| --- | --- |
| 输出等级一致性 | 所有主回复和 Skill 摘要的等级文案与愿景文档一致 |
| 高风险提示可见 | `high/critical` 输出均可见风险提示 |
| 复核门完整 | 命中复核条件时，用户必须明确确认或取消 |
| 导出前阻断有效 | 未复核的高风险内容不能直接进入最终导出链路 |
| 证据状态可辨识 | 涉及文献、事实、统计结论的输出能区分来源完整性 |

## 14. 下一步建议

完成本规范后，建议继续推进两项工作：

1. 将规范映射为具体组件方案与状态模型变更。
2. 选择 1 到 2 个高风险能力先做试点，优先建议：
   - 样本量估计
   - 统计结论解释
