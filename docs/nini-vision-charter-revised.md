# Nini 产品愿景与架构演进纲领（修订版）

> 版本：v1.1（修订草案）
> 日期：2026-03-26
> 状态：待评审
> 适用范围：产品规划、架构演进、能力准入、版本验收

---

## 第一章：文档定位

本文档用于定义 Nini 在未来 1 到 2 个版本内的产品方向、能力边界、架构演进原则与验收口径。它是战略与治理基线，不替代详细设计、接口规范、任务拆分和测试方案。

本文档适用于以下场景：

- 产品路线图评审
- 架构决策评审
- Capability / Skill 准入审查
- 版本验收与回滚判断
- 后续 PR 范围校验

若当前代码基线、核心价值承诺、目标用户范围或能力边界发生显著变化，应重新评审本文件。

---

## 第二章：当前现状基线

### 2.1 已具备的基础

Nini 当前已经具备稳定的单架构单进程底盘，核心链路为：

`Web UI -> FastAPI Gateway -> Agent Runner -> Tools / Capabilities / Skills -> Memory / Storage`

系统已支持以下基础能力：

- 本地优先的数据处理与持久化
- 受限代码执行与沙箱约束
- 数据分析核心能力与 Capability 体系
- Markdown Skill 目录与技能管理接口
- Recipe Center 类任务入口雏形
- 会话、知识、记忆与产物管理基础设施

在能力结构上，Nini 已形成 `Tools -> Capabilities -> Skills` 三层语义。数据分析阶段是当前最成熟、最可验证的核心优势。后续演进应基于现有基础增量扩展，而不是从零重建。

### 2.2 当前主要短板

当前短板主要集中在以下方面：

- 非数据分析阶段能力仍较薄弱
- 深任务缺少统一执行契约与稳定运行时
- 证据链展示、引用验证与可信输出机制不足
- 前端入口、WebSocket 主链路与部分架构叙事仍不完全一致
- 高风险科研场景的责任边界和人工复核机制不够明确

### 2.3 本文聚焦的增量

本文不试图一次性解决“科研全流程”的所有问题，也不推翻现有三层架构。本文聚焦三件事：

- 明确 V1 要做什么
- 明确哪些能力暂时不承诺
- 明确后续演进的质量门槛与风险边界

## 第三章：愿景、使命与非目标

### 3.1 愿景

Nini 的长期愿景，是成为贯穿科研工作流的 AI 研究伙伴：在保证数据主权、方法严谨、过程可审计和证据可溯源的前提下，帮助研究者更高效地完成从问题形成到结果表达的关键工作。

### 3.2 使命

Nini 的使命不是替代研究者，而是降低科研流程中的技术门槛，减少研究者在方法选择、数据处理、结构化写作和结果整理上的重复劳动，使用户把更多精力投入到研究判断与科学思考本身。

### 3.3 V1 非目标

V1 明确不承诺以下事项：

- 不承诺覆盖所有学科深水区
- 不承诺完全自动化完成投稿与修回
- 不承诺在离线模式下完成高时效文献综述
- 不承诺替代统计师、PI、伦理委员会或临床专业判断
- 不承诺通过一句话触发完整科研链路并稳定成功

对于高风险研究活动，Nini 只提供建议、草稿和辅助，不提供最终责任背书。

## 第四章：目标用户与首发场景

### 4.1 核心目标用户

V1 的核心目标用户为两类：

- 需要独立完成分析与论文撰写的硕博研究生
- 需要高频处理定量研究任务的临床或生物医学研究者

选择这两类用户的原因是：需求集中、任务链路较清晰、验证成本可控，适合作为首发优化对象。

### 4.2 次级用户

以下用户属于兼容支持对象，但不作为 V1 优先优化人群：

- 本科生
- PI / 课题组管理者
- 跨学科研究者

V1 的设计应避免为了兼容所有角色而牺牲主路径清晰度。

### 4.3 V1 首发场景

V1 聚焦三类高频且可验证的场景：

1. 数据分析到结果写作衔接
   包括统计分析、图表生成、结果段落初稿和标准化导出。
2. 文献综述草稿与证据整理
   重点是检索结果整理、主题归纳、引用清单与综述结构草稿，而不是自动替代系统综述。
3. 实验设计中的样本量估计与方案草稿
   重点是给出方法建议、参数说明和模板化草稿，而不是直接形成最终研究方案。

### 4.4 暂缓场景

以下场景不作为 V1 的主承诺能力：

- 全流程“一句话完成研究”
- 高自主度投稿期刊推荐
- 涉及患者安全或伦理判断的高风险建议
- 未经验证的跨阶段自动编排

## 第五章：核心价值承诺

Nini 的所有能力扩展都必须服从以下六项核心承诺。

### 5.1 数据主权

用户数据默认不离开本地环境。任何联网行为必须显式授权，并留下可审计记录。

### 5.2 方法严谨

统计方法、研究建议和结果描述必须说明依据。结论不得超出数据、设计与证据的支撑范围。

### 5.3 过程可审计

关键步骤、参数摘要、调用路径、失败原因和降级行为必须可追溯、可回放、可质疑。

### 5.4 证据可溯源

凡涉及事实、引用、文献总结、期刊信息等外部知识，必须标明来源。无法验证时必须明确标注不确定性。

### 5.5 渐进式复杂度

简单任务应尽量低门槛完成，复杂任务应通过参数收集、阶段拆分和进度反馈逐步推进，而不是一次性把复杂度全部压给用户。

### 5.6 人类最终负责

Nini 可以协助研究，但不能替代研究判断。高风险建议必须由用户或专业人员复核确认。

## 第六章：风险分级与输出边界

### 6.1 输出等级

Nini 的输出分为四类：

- `建议级`：仅提供方向性意见
- `草稿级`：形成可编辑初稿
- `可审阅级`：具备较完整的方法与来源信息，适合人工审阅
- `可导出级`：结构化产物达到导出标准，但仍不代表免复核使用

### 6.2 风险等级

能力按风险分为四类：

- 低风险：结构整理、模板填充、格式适配、通用写作辅助
- 中风险：文献综合、研究设计建议、引用整理
- 高风险：统计推断、样本量估计、审稿回复建议
- 极高风险：临床决策、伦理判断、因果主张背书、可能影响患者安全的建议

### 6.3 强制人工复核场景

以下内容必须显式标注“需人工复核”：

- 样本量计算
- 研究方案定稿
- 投稿回复与审稿意见处理
- 期刊适配与投稿建议
- 临床解释
- 伦理相关建议

即使这类内容达到 `可导出级`，也不得被系统表述为“最终正确答案”。

### 6.4 禁止性规则

系统不得出现以下行为：

- 在无新鲜外部来源时，输出“近 5 年研究进展综述”式强结论
- 在无权威数据库支撑时，输出强推荐式投稿期刊建议
- 在无证据支持时，把推测性内容写入最终摘要或结论段落
- 把草稿级输出伪装成已验证结论

## 第七章：V1 能力范围

### 7.1 重点阶段

V1 重点投入四个阶段：

- `⑤ 数据分析`
- `③ 实验设计`
- `② 文献调研`
- `⑥ 论文写作`

其中，数据分析继续作为核心优势深化；其余三个阶段采取限定范围、可验证交付的策略推进。

### 7.2 能力目标

#### ⑤ 数据分析

目标成熟度：`L3 / T3`

重点工作：

- 持续增强现有专用工具链
- 强化证据链与可信输出
- 强化图表、报告和标准化产物导出能力

#### ③ 实验设计

目标成熟度：`L2 / T2`

重点工作：

- 样本量估计
- 参数说明与方法依据展示
- 研究方案模板草稿

边界说明：

- 不直接替代统计方案定稿
- 不绕过人工复核

#### ② 文献调研

目标成熟度：`L2 / T2`

重点工作：

- 检索结果整理
- 主题归纳与综述结构草稿
- 引用清单与来源标注

边界说明：

- 不承诺自动完成严格系统综述
- 离线模式仅可输出草稿级结果

#### ⑥ 论文写作

目标成熟度：`L2 / T2`

重点工作：

- 结构生成
- 结果段落草稿
- 图表插入与导出衔接

边界说明：

- 不承诺自动完成高质量可直接投稿终稿
- 所有写作输出默认需人工审阅

### 7.3 基线阶段

以下阶段在 V1 中仅提供 `L1 / T1` 能力：

- `① 选题立项`
- `④ 数据采集`
- `⑦ 投稿修回`
- `⑧ 学术传播`

这些阶段主要提供方向、模板、注意事项和基础建议，不作为端到端标准产物阶段承诺。

## 第八章：能力成熟度模型

### 8.1 自动化等级

- `L1`：智能对话型支持，以知识、推理和建议为主
- `L2`：辅助编排型支持，可通过代码执行、模板和插件形成较完整草稿
- `L3`：深度原生型支持，具备专用工具链、稳定产出和较强可复现性

### 8.2 可信度等级

- `T1`：本地知识或通用推理生成的草稿级输出
- `T2`：有来源支撑、结构较完整、适合人工审阅
- `T3`：过程可审计、证据较完整、可复核且可用于标准化导出

### 8.3 组合标注原则

后续能力规划与文档描述应统一使用 `Lx / Tx` 双标签。例如：

- `L2 / T1`：自动化程度较高，但可信度仍只到草稿级
- `L3 / T3`：自动化程度和可信度都达到较高水平

这样可以避免把“能生成”误写成“能可靠交付”。

## 第九章：架构演进原则

### 9.1 三层架构保持不变

Nini 继续保持 `Tools -> Capabilities -> Skills` 三层架构，不新增第四层抽象。

原因是现有三层已足以承载以下职责：

- Tools：原子工具能力
- Capabilities：领域能力封装
- Skills：工作流模板与任务编排

当前问题主要在契约、运行时与边界不够清晰，而不在抽象层数不足。

### 9.2 增量式演进

后续演进采用以下顺序：

1. 统一契约与语义
2. 补齐运行时、观测与证据结构
3. 扩展重点场景能力
4. 最后评估跨阶段编排与生态化

### 9.3 主链路一致性

前端入口、WebSocket 主链路、MCP 暴露的能力语义应尽量对齐，避免“文档定义一套、实际运行一套”。

## 第十章：Skill 契约最小规范

### 10.1 元数据要求

每个 Skill 至少应声明以下字段：

- 名称
- 版本
- 所属阶段
- 风险等级
- 输入契约
- 输出契约
- 依赖
- 降级策略
- 可信度上限
- 是否需要人工复核

### 10.2 执行语义要求

每个步骤至少应声明：

- 步骤标识
- 输入来源
- 输出产物
- 依赖关系
- 失败终态
- 重试策略
- 是否需要人工确认

若步骤依赖联网、外部插件或高风险推理，必须额外声明授权条件与边界提示。

### 10.3 可观测性要求

Skill 运行时至少应记录：

- 步骤状态变化
- 关键参数摘要
- 工具调用摘要
- 错误原因
- 降级路径
- 最终产物列表

### 10.4 基本原则

仅有说明文档而无执行语义的 Skill，不应被描述为可执行工作流。

若后续引入 DAG 执行引擎，应优先保证状态语义与恢复语义稳定，再追求并行调度复杂度。

## 第十一章：插件与联网策略

### 11.1 联网定位

联网能力是增强项，不是默认前提。默认优先使用本地数据、本地知识和本地工具，但“离线可运行”不等于“离线可可信”。

### 11.2 离线模式边界

离线模式可用于以下任务：

- 方法建议
- 草稿生成
- 本地知识整理
- 本地数据分析与报告衔接

离线模式不应用于以下任务的强结论输出：

- 高时效文献综述
- 期刊推荐
- 外部事实校验
- 新近政策、规范、标准总结

### 11.3 联网模式门槛

只有在满足以下条件时，联网结果才能进入高可信输出：

- 来源清晰
- 权限明确
- 日志可审计
- 结果可标注来源
- 输出中明确说明证据边界

### 11.4 降级原则

插件不可用时，系统应优雅降级，但必须明确提示：

- 当前缺失了什么能力
- 当前结果降级到了什么可信度等级
- 若用户需要更高可信度，应补充什么条件

## 第十二章：实施路线图

### Phase 1：契约与边界对齐（0 到 4 周）

目标：

- 完成 Skill 最小契约规范
- 定义风险分级与输出等级
- 统一当前文档叙事与代码基线描述
- 明确主链路中的能力语义

本阶段的重点不是铺开大量新功能，而是先让后续扩展有统一边界。

### Phase 2：V1 核心能力落地（5 到 10 周）

目标：

- 增强数据分析原生能力
- 交付实验设计 L2 原型
- 交付文献综述草稿与引用整理能力
- 打通分析结果到论文初稿片段与导出的衔接

所有新增能力必须以“可验证交付”而非“覆盖更多概念”为导向。

### Phase 3：证据链与可观测强化（11 到 16 周）

目标：

- 补齐证据结构
- 完善日志结构与失败恢复
- 建立质量回放与回归基线
- 提升系统从“能做”到“可审计、可解释、可回归”的能力

### 延后事项

以下事项原则上延后到 V2 之后评估：

- 跨阶段自动编排
- 社区技能生态
- 学科定制 Skill 包
- 高自由度研究链路自动执行

---

## 第十三章：验收指标

V1 的核心验收不以“覆盖 8 个阶段”作为主指标，而以“在核心场景里是否真正更可用、更可信、更稳定”为判断标准。

### 13.1 产品指标

- 首次有效产出时间
- 首会话成功率
- 用户返工时长

### 13.2 可信性指标

- 可追溯结论占比
- 有证据输出占比
- 高风险输出人工确认率

### 13.3 工程指标

- 现有能力回归通过率
- 离线可用性
- 类型检查通过情况
- 测试覆盖率
- 发布稳定性

### 13.4 指标约束

对 V1 而言，宁可在 3 到 4 个核心场景做到稳定可用，也不应为了覆盖更多阶段而牺牲可信性和工程质量。

---

## 第十四章：治理机制

### 14.1 新能力准入

新增 Capability 或 Skill 前，必须明确：

- 阶段归属
- 风险等级
- 用户场景
- 降级策略
- 测试范围
- 输出等级上限

### 14.2 高风险能力评审

高风险能力在合并前应经过以下审查：

- 方法审查
- 边界审查
- 安全审查

### 14.3 向后兼容

所有新增能力都必须保证：

- 不破坏现有数据分析链路
- 不破坏既有接口兼容性
- 不让实验能力默认侵入稳定主路径

### 14.4 文档真实性原则

若某项能力在实现层仍是草稿或实验状态，不得在战略文档中被表述为“稳定可用”。

## 附录 A：术语表

| 术语 | 定义 |
|------|------|
| Tool | 原子函数，模型可直接调用的最小能力单元 |
| Capability | 领域能力，编排多个 Tool 完成特定业务场景 |
| Skill | 工作流模板，包含输入输出契约、步骤语义、降级策略与运行时约束 |
| Plugin | 可选扩展模块，提供联网或外部服务能力 |
| Phase | 科研流程中的一个阶段 |
| L1 / L2 / L3 | 自动化成熟度等级 |
| T1 / T2 / T3 | 输出可信度等级 |
| 风险等级 | 能力对研究结果、合规或用户决策的潜在影响等级 |
| 人工复核门 | 高风险输出进入可审阅或可导出前必须由人工确认的环节 |
