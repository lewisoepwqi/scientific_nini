## 3. ScienceClaw 优势与设计分析

### 3.1 产品定位层面的优势

1. **场景聚焦非常明确**：围绕科研研究流程而非通用助手场景，用户心智成本低。
2. **“一句话触发完整研究链路”的叙事强**：从问题到检索、分析、图表、报告导出形成端到端闭环。
3. **交付物导向明显**：强调论文/报告可用输出（图表、结论、文档），与科研用户价值直接对齐。

### 3.2 架构策略层面的优势

1. **“轻内核 + 强配置 + 技能资产”**：
   - 运行时内核尽量薄，能力更多沉淀在结构化文档与技能模板中。
   - 支持低成本扩展新任务类型（通过 Recipe/Skill 增量而非大规模改代码）。
2. **工作流模板（Recipe）作为一级抽象**：
   - 将复杂多步流程标准化为“可调用的任务模板”，降低用户提示词门槛。
3. **可信输出意识较强**：
   - 对“引用真实性、无法验证即声明不确定”有明确策略倾向。

### 3.3 对 Nini 的启示

- ScienceClaw 的核心可借鉴点不是“更复杂后端”，而是 **如何把复杂能力包装为用户可直接成功的默认体验**。
- 对 Nini 来说，关键不是补底层能力，而是升级“任务入口、证据链、交付物、观察闭环”。

## 4. Nini 当前能力与可承接基础

结合 Nini 当前文档可确认：

- 已具备稳定单架构：`Web UI -> FastAPI Gateway -> Agent Runner(ReAct) -> Skills/Tools -> Memory/Storage`。
- 已有工程基础：多模型路由、受限执行环境、SQLite + 本地文件系统持久化、MCP 接口、harness 回放评测。
- 已有能力分层：Tools/Capabilities/Skills 三层语义清晰，具备后续做任务模板编排的组织基础。

结论：

> Nini 的“工程底盘”强于多数轻量方案，短板主要在用户入口产品化与研究型任务编排资产化。

## 6. 优化迭代总策略

### 6.1 北极星目标（12~16 周）

- 新用户首次会话产出“可提交研究草稿”的成功率显著提升。
- 关键结论具备可追溯证据链与可复核方法记录。
- 深任务具备可恢复、可观测、可评测的工程能力。

### 6.2 执行原则

1. **先产品入口，后深度优化**：先提高“用得起来”的比例，再优化复杂能力。
2. **以最小增量复用现有三层架构**：不推翻 Tool/Capability/Skill 体系。
3. **每阶段都可独立上线与回滚**：降低一次性大改风险。
4. **强制指标化验收**：避免“做了很多但价值不明确”。

### 6.3 关键依赖与合规风险清单（需在 Phase A 同步确认）

- 证据来源与授权：明确检索来源、许可与引用规范，避免版权与合规风险。
- 证据持久化：来源抓取方式、存储结构与保留周期策略。
- 导出依赖：Word/PPTX/LaTeX 的库依赖与可替代方案，避免单点依赖不可用。
- 记忆生命周期：跨会话记忆的保留、删除、回滚机制与用户可见性。

## 7. 分阶段实施计划

## Phase A（0~4 周）：任务入口产品化（Recipe MVP）

### A.1 目标

把“会提问才能用好”转为“选模板即可跑通”，降低首用门槛。

### A.2 工作项

1. **Recipe Center（首批 6 个）**
   - 例如：差异表达分析、文献综述、临床方案速览、靶点验证、研究者画像、数据质量审计。
2. **意图路由增强**
   - 增加 quick task / deep task 分类器。
   - deep task 自动创建项目工作区并展示步骤进度。
3. **执行契约定义**
   - 定义 Recipe 元数据规范：输入参数、步骤 DAG、失败回退、默认输出模板。
4. **前端入口改造**
   - 首页卡片化任务入口 + 示例输入。

### A.3 验收指标

- 首次有效产出时间（TTFV）下降 ≥ 25%。
- 新用户首会话完成率提升 ≥ 30%。
- Recipe 执行失败中可自动恢复比例 ≥ 60%。

### A.3.1 指标定义与采集口径

- TTFV：首次产出可提交“研究草稿/报告/图表”的耗时；事件以“开始会话”到“首个可提交产物生成”计。
- 首会话完成率：首次会话触发 Recipe 且产出成功的占比；以新用户为样本。
- 自动恢复比例：深任务失败后自动重试并成功完成的占比；以 deep task 为样本。

### A.4 风险与回滚

- 风险：模板设计过于刚性导致适配性差。
- 回滚：Recipe 入口可灰度开关，不影响原会话入口。

---

## Phase B（5~8 周）：证据链与可信输出工程化

### B.1 目标

将“可信结论”从提示策略升级为系统约束。

### B.2 工作项（拆分为 B1/B2 两步推进）

1. **B1：证据结构与绑定**
   - Citation Graph：每条结论绑定来源（来源类型、标题、时间戳、唯一标识、抓取方式）。
   - Evidence Block：报告中输出最小证据块（来源 + 摘要 + 获取方式）。
2. **B1：METHODS 自动生成（v1）**
   - 自动记录数据来源、统计方法、参数阈值、模型版本、执行时间。
3. **B2：Claim 校验流水线**
   - 结论抽取 → 证据对齐 → 冲突检测 → 置信度评分。
4. **B2：无证据结论保护**
   - 无证据支持时，结果状态标记为“待验证”，禁止进入最终摘要。

### B.3 验收指标

- 报告中可追溯结论占比 ≥ 90%（B1）/ ≥ 95%（B2）。
- 抽样审查引用准确率 ≥ 95%（B1）/ ≥ 98%（B2）。
- 结论冲突自动识别召回率 ≥ 85%（B2）。

### B.3.1 指标定义与采集口径

- 可追溯结论：结论有唯一证据来源且可定位；以报告结论为样本。
- 引用准确率：抽样人工核验结论与来源一致性；每轮 ≥ 50 条样本。
- 冲突识别召回率：在人工标注冲突集上的召回；需维护固定基准集。

### B.4 风险与回滚

- 风险：严格校验拉高响应时延。
- 回滚：提供“草稿模式（快）/验证模式（慢）”双通道。

---

## Phase C（9~12 周）：交付物体系升级

### C.1 目标

把分析结果升级为“可直接用于论文/汇报”的标准产物。

### C.2 工作项

1. **导出管线统一**
   - 支持 Word/PPTX/LaTeX 模板化导出。
2. **图表标准化**
   - 输出统一字体、分辨率、图注、配色策略与可追溯 metadata。
3. **产物管理中心（v0）**
   - 项目级产物列表、版本号、打包下载（高级差异/回滚后置）。

### C.3 验收指标

- 导出成功率 ≥ 99%。
- 用户后期手工编辑工作量下降 ≥ 40%。
- 关键图表复用率提升 ≥ 30%。

### C.3.1 指标定义与采集口径

- 导出成功率：导出请求成功返回可用文件的比例；以导出事件为样本。
- 手工编辑下降：用户反馈 + 抽样访谈结合；至少 10 组样本对比。
- 图表复用率：同项目中复用既有图表的次数/图表总数。

### C.4 风险与回滚

- 风险：模板过多导致维护碎片化。
- 回滚：先收敛 2~3 套“高频模板”，其余走实验标记。

---

## Phase D（13~16 周）：稳定性与可观测闭环

### D.1 目标

形成可持续优化的“数据驱动迭代系统”。

### D.2 工作项

1. **全链路可观测**
   - task_id 贯穿前端事件、模型调用、工具执行、产物生成。
2. **可靠性策略**
   - 外部依赖失败重试、断线恢复、幂等执行、超时分级。
3. **质量评测体系**
   - 建立任务基准集，接入 harness 自动回放评分。
4. **成本治理**
   - 按任务类型做模型分层路由与预算阈值告警。
5. **产物管理中心高级能力（可选）**
   - 差异对比、回滚、版本关联追踪。
6. **跨会话记忆增强（可选）**
   - 基于主题/疾病/靶点的研究历史回溯与可删除策略。

### D.3 验收指标

- 深任务 P95 成功率 ≥ 90%。
- 关键流程 MTTR 下降 ≥ 40%。
- 每次发布前自动回归覆盖核心 Recipe ≥ 80%。

### D.3.1 指标定义与采集口径

- 深任务 P95 成功率：deep task 在 P95 时延范围内成功完成的比例。
- MTTR：关键故障从触发到恢复的平均时间；需绑定 incident 口径。
- 自动回归覆盖：核心 Recipe 的回放通过数/核心 Recipe 总数。

### D.4 风险与回滚

- 风险：观测维度过多导致系统复杂化。
- 回滚：分批接入关键路径指标，先保“能定位问题”。

## 8. 建议的 PR 拆分方式

说明：以下为分支名建议与 PR 标题模板，供拆分与审查使用。

为保证可审查与可回滚，建议按以下顺序拆分：

1. `docs/recipe-spec-v1`：Recipe 规范与元数据契约（文档 + schema）。
2. `feat/recipe-center-mvp`：后端路由 + 前端入口 + 3 个 Recipe。
3. `feat/citation-graph-v1`：证据结构输出与报告渲染。
4. `feat/methods-auto-writer`：METHODS 自动生成。
5. `feat/export-pipeline-v1`：Word/PPTX/LaTeX 模板导出。
6. `chore/observability-and-eval`：指标埋点与回放评测。

## 9. 近两周可执行冲刺（建议）

### Sprint-1（第 1 周）

- 产出 Recipe 规范草案（YAML/JSON Schema）。
- 实现 3 个高频 Recipe 的配置与编排接入。
- 前端新增 Recipe 入口卡片与“任务进度”基础 UI。

### Sprint-2（第 2 周）

- 引入结论-证据绑定结构（最小可用版）。
- 报告中输出 Evidence Block 与 METHODS 初版。
- 通过 harness 建立首批 20 条对比样本进行前后评测。

---

## 10. 成功判定（Definition of Success）

以下条件同时满足，视为本轮迭代成功：

1. 用户无需复杂提示词即可完成至少 3 类深任务。
2. 输出结论大部分可追溯且具备方法记录。
3. 系统具备可观测与可回归评测能力，能支撑持续优化。
4. 变更均以小 PR 分阶段落地，任何阶段可独立回滚。

---

## 11. 附录：执行注意事项

- 新增功能优先复用现有 Tool/Capability/Skill 体系，不做无关重构。
- 任何高风险操作（批量删除、重置历史、数据清理）必须人工确认。
- 推进顺序建议：`规范定义 → MVP接入 → 观测与评测 → 扩展与优化`。
