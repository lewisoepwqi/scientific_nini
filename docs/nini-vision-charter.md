本文档定义 Nini 在未来 1～2 个版本内的产品方向、能力边界、架构演进原则与验收口径。它是**战略与治理基线**，不替代详细设计、接口规范、任务拆分和测试方案。

## 第一章：现状基线与愿景

### 1.1 当前基线

Nini 当前已具备稳定的单架构单进程底盘：

```
Web UI → FastAPI Gateway → Agent Runner (ReAct)
       → Tools / Capabilities / Skills → Memory / Storage
```

**已具备的基础**：
- 本地优先的数据处理与持久化
- 受限代码执行与沙箱约束（Python + R）
- Capability 体系已定义 10 个能力：6 个核心数据分析类（差异分析、相关性、回归、数据探索、清洗、可视化，其中 5 个可执行）+ 4 个全流程扩展类（报告生成、科研文章初稿、引用管理、研究规划，均为元数据定义阶段）
- Markdown Skill 体系已运行：扫描/注册/触发/启用禁用管理、YAML frontmatter 元数据、自然语言匹配、正文注入 runtime context、工具权限白名单；已有 `root-analysis`、`article-draft`、`publication_figure` 等实际技能
- 多模型路由（OpenAI、Anthropic、Ollama、Moonshot、智谱、DeepSeek、DashScope/阿里百炼、MiniMax 等 9 个提供商）
- 会话、知识、记忆与产物管理基础设施
- 三层提示词架构（System Prompt / Runtime Context / Tool Schema）

**当前主要短板**：
- 非数据分析阶段能力仍较薄弱（代码中已有 `article_draft`、`citation_management`、`research_planning` 等 Capability 定义，但多数尚未可执行）
- Markdown Skills 已有提示词驱动的工作流描述，但缺少结构化执行契约（步骤 DAG、显式依赖、失败恢复、降级声明、可信度标注、人工复核门）——目前依赖 LLM 自主遵从提示词，缺乏运行时引擎的确定性保障
- 证据链展示、引用验证与可信输出机制不足
- 高风险科研场景的责任边界和人工复核机制不够明确
- 插件/联网能力框架缺失

### 1.2 愿景

> **从选题到发表，让每位研究者都有一位懂方法、守规范、数据安全的 AI 科研伙伴。**

Nini 的长期愿景，是成为贯穿科研工作流的 AI 研究伙伴：在保证数据主权、方法严谨、过程可审计和证据可溯源的前提下，帮助研究者更高效地完成从问题形成到结果表达的关键工作。

### 1.3 使命

Nini 的使命不是替代研究者，而是**降低科研流程中的技术门槛**——减少研究者在方法选择、数据处理、结构化写作和结果整理上的重复劳动，使用户把更多精力投入到研究判断与科学思考本身。

### 1.4 定位升级：从「数据分析」到「科研全流程」

#### 1.4.1 过去：科研数据分析 AI 助手

Nini 最初聚焦于科研数据分析——用户上传数据，Agent 自动完成统计检验、可视化、报告生成。这一定位在数据分析阶段已建立扎实的工程底盘和能力深度。

#### 1.4.2 现在：科研全流程 AI 伙伴

科研不只是数据分析。一个完整的研究项目从选题到发表，经历 8 个核心阶段。Nini 的定位从「分析工具」升级为「研究伙伴」，意味着：

- **覆盖广度**：从单一数据分析阶段，扩展到选题、文献、设计、采集、分析、写作、投稿、传播全流程。
- **交互深度**：从「工具调用」升级为「研究协作」——Nini 不仅执行任务，还能参与方案讨论、提供方法建议、审查研究设计。
- **价值定位**：从「提高分析效率」升级为「提升研究质量」——确保方法严谨、证据可溯、结论可信。

#### 1.4.3 不变的根基

升级不意味着推翻，以下根基不动摇：

- **数据分析仍是核心优势**：技术护城河，新阶段在此基础上向两端延伸。
- **本地优先架构**：用户数据默认不离开本地环境，联网行为需明确授权。
- **单架构单进程**：保持轻量部署，不引入分布式复杂度。

### 1.5 V1 非目标

V1 明确**不承诺**以下事项，避免过度承诺导致交付失真：

- 不承诺覆盖所有学科深水区（保持学科中立但不深入特定领域）
- 不承诺完全自动化完成投稿与修回
- 不承诺在离线模式下完成高时效文献综述
- 不承诺替代统计师、PI、伦理委员会或临床专业判断
- 不承诺通过一句话触发完整科研链路并稳定成功
- 不承诺跨阶段自动编排、社区技能生态、学科定制 Skill 包

对于高风险研究活动，Nini 只提供建议、草稿和辅助，不提供最终责任背书。

## 第二章：核心价值承诺

Nini 的所有能力扩展都必须服从以下七项核心承诺——无论扩展到哪个研究阶段，都不可打折。

| # | 承诺 | 含义 | 验证方式 |
|---|------|------|----------|
| 1 | **数据主权** | 用户数据默认不离开本地，联网行为需明确授权并留下审计记录 | 网络请求审计日志；离线模式可用性测试 |
| 2 | **方法严谨** | 方法选择有据可依，结论不超出数据与证据支撑范围 | 方法选择需记录决策理由；结论必须标注置信度 |
| 3 | **过程可审计** | 关键步骤、参数摘要、调用路径、失败原因和降级行为可追溯、可回放、可质疑 | 完整的执行日志；Skill 运行时事件流 |
| 4 | **证据可溯源** | 引用标注来源，无法验证时明确声明不确定性 | 证据链完整性检查；不确定性标签机制 |
| 5 | **学科中立** | 不预设学科假设，适用于任何研究领域 | 跨学科测试用例覆盖；无硬编码学科逻辑 |
| 6 | **渐进式复杂度** | 简单任务立即可用，复杂任务按需深入 | 首次使用无需配置；高级功能可选激活 |
| 7 | **人类最终负责** | Nini 协助研究但不替代研究判断；高风险建议必须由用户或专业人员复核确认 | 高风险输出强制人工复核门；输出等级标注 |

## 第三章：目标用户与首发场景

### 3.1 V1 核心目标用户

V1 优先服务两类用户——需求集中、任务链路清晰、验证成本可控：

| 用户类型 | 典型场景 | 核心诉求 |
|----------|----------|----------|
| **硕博研究生** | 独立完成分析与论文撰写 | 分析效率、写作辅助、方法严谨性 |
| **临床/生物医学研究者** | 高频定量研究任务 | 快速产出、合规性、可解释结果 |

### 3.2 次级用户（兼容但不优先优化）

| 用户类型 | 说明 |
|----------|------|
| 本科生 | 方法入门指导、规范模板 |
| PI / 课题组长 | 质量把控、流程标准化 |
| 跨学科研究者 | 学科中立的方法推荐 |

V1 的设计应避免为了兼容所有角色而牺牲主路径清晰度。

### 3.3 V1 首发场景

V1 聚焦三类高频且可验证的场景：

1. **数据分析到结果写作衔接**：统计分析、图表生成、结果段落初稿和标准化导出。
2. **文献综述草稿与证据整理**：检索结果整理、主题归纳、引用清单与综述结构草稿。（不是自动替代系统综述）
3. **实验设计中的样本量估计与方案草稿**：方法建议、参数说明和模板化草稿。（不是直接形成最终研究方案）

### 3.4 暂缓场景

以下场景不作为 V1 主承诺能力：

- 全流程"一句话完成研究"
- 高自主度投稿期刊推荐
- 涉及患者安全或伦理判断的高风险建议
- 未经验证的跨阶段自动编排

## 第四章：风险分级与输出边界

### 4.1 输出等级

Nini 的输出分为四个等级，明确「能生成」与「可信可用」的边界：

| 等级 | 名称 | 定义 | 用户预期 |
|------|------|------|----------|
| **O1** | 建议级 | 方向性意见，仅供参考 | 需要用户独立判断 |
| **O2** | 草稿级 | 可编辑初稿，结构较完整 | 需要用户修改和补充 |
| **O3** | 可审阅级 | 方法与来源信息较完整，适合人工审阅 | 需要专业人员复核 |
| **O4** | 可导出级 | 结构化产物达到导出标准 | 仍需人工终审，不代表免复核 |

### 4.2 风险等级

能力按对研究结果、合规或用户决策的潜在影响分为四级：

| 风险等级 | 定义 | 示例 |
|----------|------|------|
| **低** | 错误主要影响表达和效率，不直接影响研究判断 | 结构整理、格式适配、通用写作辅助 |
| **中** | 错误会影响草稿质量或资料完整性 | 文献归纳、引用整理、综述结构 |
| **高** | 错误会影响研究方法、统计判断或投稿策略 | 样本量估计、统计推断解释、审稿回复 |
| **极高** | 错误可能影响患者安全、伦理合规或重大研究结论 | 临床建议、伦理判断、因果主张背书 |

若对风险级别存在争议，按更高等级处理。

### 4.3 强制人工复核场景

以下内容必须显式标注"需人工复核"并触发人工复核门：

- 样本量计算与关键参数推荐
- 研究方案定稿
- 统计结论的最终解释
- 投稿回复与审稿意见处理
- 期刊适配与投稿建议
- 临床解释与伦理相关建议
- 即将进入最终摘要、导出文档或外部发送材料的内容

即使这类内容达到 O4（可导出级），也不得被系统表述为"最终正确答案"。

### 4.4 禁止性规则

系统不得出现以下行为：

1. 在无新鲜外部来源时，输出"近 X 年研究进展综述"式强结论
2. 在无权威数据库支撑时，输出强推荐式投稿期刊建议
3. 在无证据支持时，把推测性内容写入最终摘要或结论段落
4. 把草稿级输出伪装成已验证结论
5. 在证据不足时给出确定性结论
6. 在离线模式下伪装输出高时效外部事实
7. 自动跳过人工复核门继续生成最终结论
8. 将低可信度输出提升为高可信度而不提示依据

## 第五章：科研全流程能力模型

### 5.1 八大研究阶段

```
科研全流程闭环
═══════════════════════════════════════════════════════════

  ① 选题立项 ──▶ ② 文献调研 ──▶ ③ 实验设计 ──▶ ④ 数据采集
       ▲                                              │
       │                                              ▼
  ⑧ 学术传播 ◀── ⑦ 投稿修回 ◀── ⑥ 论文写作 ◀── ⑤ 数据分析
```

| 阶段 | 代号 | 核心活动 | 典型产出物 |
|------|------|----------|------------|
| ① 选题立项 | `topic_selection` | 研究方向探索、可行性评估、文献缺口识别 | 研究问题陈述、可行性报告 |
| ② 文献调研 | `literature_review` | 文献检索、筛选、综述撰写、知识图谱构建 | 文献综述、参考文献库 |
| ③ 实验设计 | `experiment_design` | 方案设计、样本量估计、随机化、伦理审查准备 | 研究方案、统计分析计划（SAP） |
| ④ 数据采集 | `data_collection` | 问卷设计、数据录入模板、质控方案 | 数据采集工具、质控报告 |
| ⑤ 数据分析 | `data_analysis` | 数据清洗、统计检验、建模、可视化 | 统计报告、图表、数据集 |
| ⑥ 论文写作 | `paper_writing` | 论文结构、初稿撰写、图表排版、参考文献 | 论文初稿（Markdown/Word/LaTeX） |
| ⑦ 投稿修回 | `submission` | 期刊选择、格式适配、审稿意见回复 | 投稿版本、回复信 |
| ⑧ 学术传播 | `dissemination` | 学术报告准备、海报制作、科普文章 | 演示文稿、海报、通俗摘要 |

### 5.2 能力成熟度双轴模型

单一维度无法准确描述能力状态——「能做」不等于「可信」。采用 **Lx（自动化等级） / Tx（可信度等级）** 双轴标注。

#### 5.2.1 自动化等级（Lx）

| 等级 | 名称 | 定义 | Agent 行为特征 |
|------|------|------|----------------|
| **L1** | 智能对话 | 给出专业建议，不直接操作数据/文件 | 基于知识库和推理提供建议 |
| **L2** | 辅助编排 | 通过代码执行 + 提示词引导完成任务 | 调用代码执行、生成结构化文档、调用插件 |
| **L3** | 深度原生 | 有专用工具链，端到端可执行并产出标准化产物 | 专用 Tool 链、标准化输出模板 |

#### 5.2.2 可信度等级（Tx）

| 等级 | 名称 | 定义 | 对应输出等级 |
|------|------|------|-------------|
| **T1** | 草稿级 | 本地知识或通用推理生成，仅供参考 | O1 建议级、O2 草稿级 |
| **T2** | 可审阅级 | 有来源支撑、结构较完整，适合人工审阅 | O3 可审阅级 |
| **T3** | 可复核级 | 过程可审计、证据较完整、可用于标准化导出 | O4 可导出级 |

#### 5.2.3 双轴组合的意义

```
可信度与自动化的关系
═══════════════════════════════════════════════════════════

  T3 ┤                              ● ⑤ 数据分析 (L3/T3)
     │
  T2 ┤         ● ②③⑥ (L2/T2)
     │         文献/设计/写作
  T1 ┤  ● ①④⑦⑧ (L1/T1)
     │  基线阶段
     └──┬────────┬────────┬──▶ 自动化
       L1       L2       L3

  关键洞察：
  ├── L2/T1 = 能自动生成，但可信度只到草稿级（不可误导用户）
  ├── L3/T3 = 自动化和可信度都高（当前仅数据分析达到）
  └── 升级路径不是单维的，需要同时提升两个轴
```

### 5.3 V1 能力矩阵

| 阶段 | V1 目标 | 风险等级 | 说明 |
|------|---------|----------|------|
| ⑤ 数据分析 | **L3 / T3** | 高 | 核心优势持续深化；强化证据链与可信输出 |
| ③ 实验设计 | **L2 / T2** | 高 | 样本量估计、方案模板；不替代统计方案定稿 |
| ② 文献调研 | **L2 / T2** | 中 | 综述草稿、引用清单；不承诺严格系统综述 |
| ⑥ 论文写作 | **L2 / T2** | 中 | 结构生成、结果段落草稿；默认需人工审阅 |
| ① 选题立项 | **L1 / T1** | 低 | 方向建议、可行性评估 |
| ④ 数据采集 | **L1 / T1** | 低 | 问卷建议、样本策略 |
| ⑦ 投稿修回 | **L1 / T1** | 高 | 期刊推荐、审稿回复建议；仅建议级 |
| ⑧ 学术传播 | **L1 / T1** | 低 | 报告提纲、海报要素 |

#### V1 重点阶段详述

**⑤ 数据分析（L3 / T3）**：
- 持续增强现有数据分析类 Capability 专用工具链（差异分析、相关性、回归、清洗、可视化等）
- 强化证据链与可信输出（对接 ScienceClaw 迭代计划 Phase B）
- 增强报告导出模板（Word/LaTeX）

**③ 实验设计（L2 / T2）**：
- 样本量估计（Python Tool，功效分析计算）
- 参数说明与方法依据展示
- 研究方案模板草稿（Markdown Skill）
- **边界**：不直接替代统计方案定稿；不绕过人工复核

**② 文献调研（L2 / T2）**：
- 检索结果整理（联网插件可选增强）
- 主题归纳与综述结构草稿（Markdown Skill）
- 引用清单与来源标注
- **边界**：不承诺自动完成严格系统综述；离线模式仅输出 T1 草稿级结果

**⑥ 论文写作（L2 / T2）**：
- 结构生成、结果段落草稿
- 图表插入与导出衔接
- **边界**：不承诺自动完成可直接投稿终稿；所有写作输出默认需人工审阅

### 5.4 长期演进路线

```
能力演进时间线
═══════════════════════════════════════════════════════════

  V1（近期）             V2（中期）             V3（长期）
  ┌───────────┐         ┌───────────┐         ┌───────────┐
  │ ⑤ L3/T3   │         │ ⑤ L3/T3+  │         │ 全阶段     │
  │ ②③⑥ L2/T2│────────▶│ ②③⑥ L3/T3│────────▶│ L3/T3     │
  │ ①④⑦⑧ L1/T1│        │ ①④⑦⑧ L2/T2│        │ 跨阶段    │
  └───────────┘         └───────────┘         │ 智能编排   │
                                               └───────────┘

  V1：全阶段 L1 基线 + 重点阶段 L2 突破
  V2：L2→L3 升级 + 全阶段 L2 覆盖
  V3：跨阶段自动编排 + 社区生态
```

### 5.5 可信度与风险联动规则

Skill 的输出可信度不得超过其声明的 `trust_ceiling`。

| 风险等级 | 默认可信度上限 | 无人工复核可进入最终摘要？ |
|----------|----------------|---------------------------|
| 低 | T2 | 允许 |
| 中 | T2 | 视证据完整性而定 |
| 高 | T2 | 默认不允许 |
| 极高 | T1 或 T2 | 不允许 |

## 第六章：架构演进方案

### 6.1 三层架构演进

#### 6.1.1 核心决策：三层不加新层

Nini 采用 **Tools → Capabilities → Skills** 三层架构。升级过程中不引入新的抽象层（如 Recipe），而是激活并演进现有各层。

| 层级 | 当前状态 | 演进方向 |
|------|----------|----------|
| **Skills** | 提示词驱动已运行，缺结构化执行契约 | 演进为契约驱动的工作流模板层（吸收 Recipe 理念） |
| **Capabilities** | 6 个数据分析类 | 泛化为多阶段领域能力层 |
| **Tools** | 统计/图表/代码类 | 按需扩展新领域原子工具 |

**决策理由**：ScienceClaw 的 Recipe 概念本质上就是 Skills 层应承载的功能。Markdown Skills 体系已在运行（扫描、注册、触发、元数据管理），缺失的不是抽象层，而是执行契约与运行时引擎。新建第四层会导致 Skill vs Recipe 概念模糊、边界不清，增加维护成本但不带来新能力。正确的演进方向是**在现有 Markdown Skill 基础上增量补齐执行契约**（步骤 DAG、降级策略、可信度声明、人工复核门），而非推翻重来。

### 6.2 Skill 执行契约概要

> 完整规范见 `docs/skill-contract-spec.md`

#### 6.2.0 与现有 Markdown Skill 的关系

现有 Markdown Skills（如 `root-analysis`、`article-draft`）已具备元数据（name, description, category, tags, allowed-tools）和自然语言工作流描述，通过提示词注入引导 LLM 执行。这套机制对 **L2 级别能力**（辅助编排）足够——LLM 阅读提示词后自主编排工具完成任务。

执行契约是在此基础上的**增量演进**，为以下场景服务：
- **L3 级别能力**：需要确定性步骤执行，不能完全依赖 LLM 遵从度
- **高风险场景**：需要强制人工复核门、可信度天花板、证据要求
- **可观测性**：需要结构化事件流支撑进度追踪和失败恢复

演进路径：现有 SKILL.md frontmatter **扩展**新字段（risk_level, trust_ceiling, steps 等），而非替换现有格式。纯提示词驱动的低风险 Skill 可以继续按现有方式运行，无需强制升级。

#### 6.2.1 最小元数据

每个 Skill 至少声明：

| 字段 | 必填 | 说明 |
|------|------|------|
| `name` | 是 | 内部唯一标识，`snake_case` |
| `display_name` | 是 | 展示名称 |
| `version` | 是 | 契约版本 |
| `phase` | 是 | 所属研究阶段 |
| `risk_level` | 是 | `low / medium / high / critical` |
| `trust_ceiling` | 是 | 输出可信度上限：`T1 / T2 / T3` |
| `requires_human_review` | 是 | 是否存在强制人工复核点 |
| `input_schema` | 是 | 输入参数契约 |
| `output_schema` | 是 | 输出产物契约 |
| `steps` | 是 | 步骤列表（DAG） |
| `fallback` | 否 | 全局降级策略 |
| `required_plugins` / `optional_plugins` | 否 | 插件依赖 |
| `observability` | 是 | 运行时记录要求 |

#### 6.2.2 步骤类型

| 类型 | 说明 |
|------|------|
| `tool` | 调用原子 Tool |
| `capability` | 调用 Capability 编排 |
| `review_gate` | 人工复核门——需用户确认后才可继续 |

#### 6.2.3 降级与失败终态

Skill 必须区分三类结果：

- **正常完成**：所有步骤成功，输出达到声明可信度
- **降级完成**：部分步骤降级（如插件不可用），输出可信度同步下调，且必须提示用户
- **失败终止**：关键步骤失败且无降级路径

降级不是"无提示地继续"。降级后必须：
1. 同步降低可信度等级
2. 明确告知用户缺失了什么能力
3. 说明当前结果降级到了什么等级
4. 对高风险输出，缺证据时优先阻断而非默认继续

#### 6.2.4 人工复核门

以下任一情况出现时，应在 Skill 步骤中插入 `review_gate`：

- 输出属于高风险或极高风险
- 关键参数具有明显方法学后果
- 输出将进入 O4（可导出级）或最终摘要
- 外部来源存在冲突或证据不足

人工复核门至少记录：待确认内容摘要、风险说明、用户确认结果、确认时间。

#### 6.2.5 可观测性

每个 Skill 运行时至少输出以下事件：

`skill_started` → `step_started` → `step_completed / step_failed / step_degraded` → `review_requested` → `review_completed` → `skill_completed`

每条事件至少包含：`skill_name`、`skill_version`、`step_id`、`timestamp`、`status`、`input_summary`、`output_summary`、`trust_level`。

记录摘要而非原始敏感数据，保证可追溯但不扩大隐私暴露面。

#### 6.2.6 基本原则

仅有说明文档而无执行语义的 Skill，不应被表述为"可执行工作流"。若文档定义一套、实际运行一套，必须修正对齐。

### 6.3 Capability 泛化设计

#### 6.3.1 规划新增字段

当前 Capability dataclass 的字段为：`name`、`display_name`、`description`、`icon`、`required_tools`、`suggested_workflow`、`is_executable`、`execution_message`、`executor_factory`。代码中已有 4 个全流程扩展 Capability（`article_draft`、`citation_management`、`research_planning`、`report_generation`），说明 Capability 体系已开始向多阶段延伸。

规划新增以下字段以支持阶段化分类和风险标注：

```python
@dataclass
class Capability:
    # ... 现有字段保持不变 ...

    # 规划新增：所属研究阶段
    phase: str = "data_analysis"

    # 规划新增：工作流类型
    workflow_type: str = "tool_orchestration"

    # 规划新增：风险等级
    risk_level: str = "low"
```

> **注意**：`risk_level`、`trust_ceiling`、`requires_human_review` 等治理类字段在 Capability 层仅做轻量标注；完整的风险治理语义（降级策略、人工复核门、可观测事件等）由 Skill 层的执行契约承载。Capability 层保持轻薄元数据定位，避免与 Skill 层职责重叠。

#### 6.3.2 工作流类型

| 类型 | 模式 | 适用阶段 | 代表能力 |
|------|------|----------|----------|
| `tool_orchestration` | 多工具顺序/并行编排 | ⑤ 数据分析 | 差异分析、回归（现有） |
| `search_synthesis` | 检索 → 过滤 → 综合 | ② 文献调研 | 文献综述、知识图谱 |
| `reasoning_template` | 推理 → 计算 → 文档生成 | ③ 实验设计 | 样本量估计、随机化方案 |
| `generative_iterate` | 生成 → 评审 → 修订循环 | ⑥⑦ 写作/投稿 | 论文初稿、审稿回复 |
| `hybrid` | 以上多种混合 | 跨阶段 | 跨阶段复合工作流 |

### 6.4 混合实现策略

根据能力特征选择实现路线：

```
实现路线决策树

  需要精确数值计算？

```

#### 6.5.1 定位

联网能力是**增强项，不是默认前提**。默认优先使用本地数据和工具。但"离线可运行"不等于"离线可可信"——离线环境下的输出可信度可能需要下调。

#### 6.5.3 离线模式边界

**离线可用**：方法建议、草稿生成、本地知识整理、本地数据分析与报告。

**离线不应产出强结论**：高时效文献综述、期刊推荐、外部事实校验、新近政策/规范总结。

#### 6.5.4 联网模式门槛

联网结果进入高可信输出的条件：来源清晰、权限明确、日志可审计、结果可标注来源、输出中明确说明证据边界。

#### 6.5.5 降级原则

插件不可用时，必须明确提示三件事：
1. 当前缺失了什么能力
2. 当前结果降级到了什么可信度等级
3. 若用户需要更高可信度，应补充什么条件

### 7.1 总体策略

V1 的实施遵循以下原则：
- **先契约后功能**：先统一边界和规范，再铺开新能力

## 第八章：治理与演进原则

### 8.1 能力准入标准

新增任何 Capability 或 Skill 之前，必须满足：

| # | 准入条件 | 验证方式 |
|---|----------|----------|
| 1 | **阶段归属明确**：属于八大研究阶段之一 | 标注 `phase` 字段 |
| 2 | **风险等级已声明** | 标注 `risk_level` 字段 |
| 3 | **输出等级上限已声明** | 标注 `trust_ceiling` 字段 |
| 4 | **不违反核心价值承诺** | 代码审查 + 安全评估 |
| 5 | **离线可用或优雅降级** | 离线环境测试 |
| 6 | **不破坏现有能力** | 回归测试全部通过 |
| 7 | **有测试覆盖** | 测试覆盖率检查 |
| 8 | **有用户场景** | 至少一个具体使用场景 |

### 8.2 高风险能力评审

> 完整规范见 `docs/high-risk-capability-review.md`

高风险（`high`）和极高风险（`critical`）能力在合并前必须经过三维评审：

**方法评审**：方法选择有据、参数解释充分、无误导性默认值、区分建议/草稿/可审阅输出。

**边界评审**：声明非目标和禁止性输出、高风险输出前有人工复核门、不把草稿伪装成最终结论。

**安全与合规评审**：数据不会意外离开本地、联网来源合规、证据可追溯、日志记录摘要而非敏感原文。

**上线门槛**（高风险能力额外要求）：
- 至少有一组正向样例和一组失败/降级样例
- 文档、提示词和实现边界保持一致
- 对 `critical` 级别，若无法提供稳定人工复核链路，不得上线为默认可用能力

**回滚条件**——出现以下任一情况时立即回滚/降级：
- 产出误导性研究结论
- 人工复核门被绕过
- 证据链无法对齐输出结论
- 用户可能将结果误用为临床或伦理决策依据

### 8.3 向后兼容策略

| 原则 | 具体措施 |
|------|----------|
| **API 兼容** | 现有 HTTP/WebSocket 接口不改变签名；新功能通过新端点暴露 |
| **数据兼容** | 现有会话数据格式不变；新字段使用 Optional 默认值 |
| **行为兼容** | 数据分析默认行为不因升级而改变；新阶段能力默认不侵入现有流程 |
| **前端兼容** | 现有界面保持不变；新阶段入口以可选面板/Tab 形式新增 |

### 8.4 文档真实性原则

若某项能力在实现层仍是草稿或实验状态，不得在文档中被表述为"稳定可用"。能力状态必须与代码基线一致。

### 8.5 质量门禁

| 门禁 | 标准 | 级别 |
|------|------|------|
| 单元测试 | 新增代码覆盖率 ≥ 80% | 阻断合并 |
| 回归测试 | 现有测试全部通过 | 阻断合并 |
| 类型检查 | mypy 无新增错误 | 阻断合并 |
| 格式规范 | black + lint 通过 | 阻断合并 |
| 离线测试 | 断网环境下核心功能可用 | 阻断发布 |
| 安全审查 | 无敏感信息泄露、沙箱策略完整 | 阻断发布 |
| 高风险评审 | 高/极高风险能力通过三维评审 | 阻断发布 |

### 8.6 验收指标

V1 的核心验收不以"覆盖 8 个阶段"为主指标，而以"在核心场景是否真正更可用、更可信、更稳定"为判断标准。

**产品指标**：
| 指标 | V1 目标 | 度量方式 |
|------|---------|----------|
| 首次有效产出时间（TTFV） | 较现状下降 ≥ 25% | 用户测试 |
| 首会话成功率 | ≥ 60% | 埋点统计 |
| 用户返工时长 | 较现状减少 | 用户反馈 |

**可信性指标**：
| 指标 | V1 目标 | 度量方式 |
|------|---------|----------|
| 可追溯结论占比 | ≥ 80% | 产出物审计 |
| 高风险输出人工确认率 | 100% | 系统日志 |

**工程指标**：
| 指标 | V1 目标 | 度量方式 |
|------|---------|----------|
| 现有能力回归通过率 | 100% | CI 自动化 |
| 新增代码测试覆盖率 | ≥ 80% | CI 自动化 |
| 离线可用性 | 核心功能 100% | 离线环境测试 |

## 附录 A：术语表

| 术语 | 定义 |
|------|------|
| Tool | 原子函数，模型可直接调用的最小能力单元 |
| Capability | 领域能力，编排多个 Tool 完成特定业务场景 |
| Skill | 工作流模板，包含步骤 DAG、输入/输出契约、降级策略、风险声明与运行时约束 |
| Plugin | 可选扩展模块，提供联网或外部服务能力 |
| Phase | 科研流程中的一个阶段（共 8 个） |
| L1 / L2 / L3 | 自动化成熟度等级（智能对话 / 辅助编排 / 深度原生） |
| T1 / T2 / T3 | 输出可信度等级（草稿级 / 可审阅级 / 可复核级） |
| O1 / O2 / O3 / O4 | 输出等级（建议级 / 草稿级 / 可审阅级 / 可导出级） |
| 风险等级 | 能力对研究结果、合规或用户决策的潜在影响等级（低 / 中 / 高 / 极高） |
| 人工复核门 | 高风险输出进入可审阅或可导出前必须由人工确认的环节 |
| DAG | 有向无环图，描述 Skill 步骤间的依赖关系 |
| 降级 | 功能不完全可用时的备选行为（需明确提示用户） |
| TTFV | Time to First Value，首次产出有效产物的时间 |
| trust_ceiling | Skill 声明的输出可信度上限，运行时不得超过 |
| review_gate | Skill 步骤类型之一，强制人工确认后才可继续执行 |
