## Why

当前用户升级 Nini 需要离开软件、重新下载安装包并手动安装，流程成本高且容易导致用户停留在旧版本。Nini 已进入发布后维护阶段，需要提供软件内版本检查、下载与半自动静默升级能力，降低升级阻力并提升安全修复触达率。

## What Changes

- 新增应用内更新能力：Nini 可从服务器检查最新版本、展示更新说明、下载 Windows 安装包并校验完整性。
- 新增半自动升级流程：用户确认后，Nini 启动独立 updater，当前进程退出，updater 等待文件锁释放后静默运行 NSIS 安装器并重启 Nini。
- 新增更新状态 API：前端可查询检查、下载、校验、准备安装、升级失败等状态。
- 新增前端更新入口：在启动后低频自动检查更新，并在设置或全局提示中支持手动检查、下载和立即升级。
- 新增发布元数据约定：服务器通过 manifest 描述版本、渠道、安装包 URL、大小、SHA256、发布时间、更新说明和重要更新提示。
- 新增安全约束：默认只允许 HTTPS 更新源；无域名的内网部署可显式允许私有 IP/localhost 的 HTTP 更新源；安装包必须通过 SHA256 与 Authenticode 签名校验；开发环境不执行真实升级。
- 不包含二进制差分更新、跨平台更新、完全无感后台升级和自动回滚。

## Capabilities

### New Capabilities
- `in-app-updater`: 覆盖 Nini 应用内版本检查、更新包下载校验、用户确认后的半自动静默安装与失败处理行为。

### Modified Capabilities

无。

## Impact

- 后端：新增 `src/nini/update/` 服务模块与 `src/nini/api/update_routes.py`，并在 FastAPI 应用中注册更新 API。
- Windows 桌面/打包：新增独立 `nini-updater.exe` 入口，调整 `nini.spec`、`build_windows.bat` 和 NSIS 安装流程以支持覆盖安装、日志与重启。
- 前端：新增更新状态 store、更新提示/对话框/设置入口，并在应用初始化时执行低频版本检查。
- 配置：新增更新源 URL、更新渠道、自动检查开关与检查间隔配置，默认稳定渠道。
- 发布系统：构建流程需要生成或校验 manifest、安装包 SHA256、签名策略，并保留发布包下载路径。
- 测试：新增版本比较、manifest 校验、下载校验、apply 安全限制、API 路由、前端状态与打包升级烟测。
- 风险与回滚：升级失败不得删除用户数据；下载或校验失败保留当前版本；updater 启动失败时当前 Nini 继续运行；如新版本异常，可重新安装上一版本安装包回滚。
