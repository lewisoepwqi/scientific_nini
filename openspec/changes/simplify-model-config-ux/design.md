## Context

当前 `ModelConfigPanel` 采用三 Tab 结构（凭证管理 / 路由策略 / 优先级），暴露了 API Key、Base URL、6 种用途路由、拖拽优先级降级链等开发者概念。目标用户是科研人员，无法独立完成配置。产品即将对外分发并引入付费，需要：

1. 消除配置门槛（首次启动无需配置即可体验）
2. 收窄供应商为国内主流 4 家 + 本地
3. 隐藏所有路由/优先级概念
4. 用本地时限试用降低付费前门槛

约束：本地优先，无账号系统，无云端用户标识。

## Goals / Non-Goals

**Goals:**
- 科研用户能在 3 步内完成模型配置（选供应商 → 粘贴密钥 → 测试保存）
- 首次启动即可发送消息（试用模式自动激活）
- 支持 4 个供应商：DeepSeek / 智谱 GLM / 阿里百炼 / Ollama（本地）
- 模型列表从供应商 API 动态拉取，不硬编码
- 标题生成自动使用廉价模型，对用户透明

**Non-Goals:**
- 多供应商并发降级（本次删除此功能）
- 云端用户追踪或账号系统
- 付费托管服务（内置服务）
- 支持超过 4 个供应商
- 用途级路由配置（对用户暴露）

## Decisions

### D1：单一激活供应商，切换即全量生效

**选择**：同一时间只有一个供应商处于激活状态，所有请求（含标题生成）均走该供应商。切换供应商后立即生效。

**理由**：多供应商并发 + 降级链对科研用户造成"我配置了 A，为什么实际用的是 B"的困惑。单一激活逻辑透明、可预期，与用户心智模型一致。

**放弃的方案**：保留降级链但隐藏 UI → 行为不透明，debug 困难，且与"单一激活"语义冲突。

---

### D2：标题生成廉价模型——后端按偏好顺序从动态列表中自动选择

**选择**：后端维护每个供应商的"廉价模型偏好顺序"常量，在动态拉取的可用模型列表中依次匹配第一个存在的模型用于标题生成。用户不可见、不可配置。

```python
TITLE_MODEL_PREFERENCE = {
    "deepseek":   ["deepseek-chat"],
    "zhipu":      ["glm-4-flash", "glm-4-air", "glm-4"],
    "dashscope":  ["qwen-turbo", "qwen-plus"],
    "ollama":     None,  # 与主模型相同
}
```

**理由**：不硬编码模型是否存在（由动态列表决定），但偏好顺序是稳定的业务知识，无需暴露给用户。

**放弃的方案**：让用户在"高级设置"里选标题模型 → 增加认知负担，违背简化目标。

---

### D3：试用模式——纯本地时限，内嵌密钥

**选择**：首次发消息时记录安装日期到本地 SQLite（`app_settings.trial_install_date`），14 天内使用内嵌的试用密钥（DeepSeek API Key，硬编码在 `config.py`，可通过环境变量覆盖）。超期后阻断并引导配置。

**理由**：零后端依赖，立刻可上线，符合本地优先原则。早期阶段用户量小，滥用风险可控（可随时轮换内嵌密钥）。

**风险接受**：用户可通过清除本地数据重置试用期，视为可接受的早期阶段权衡。

**放弃的方案**：
- 设备 ID + 后端追踪 → 需要额外基础设施，延迟上线
- 激活码制 → 增加用户操作步骤，降低转化率

---

### D4：动态模型列表 + 前端友好名称映射层

**选择**：密钥验证成功后，调用 `GET /api/models/{provider_id}/available` 实时拉取模型列表。前端维护一个静态映射表（`MODEL_DISPLAY_NAMES`），将已知模型名翻译为友好名称 + 描述。未在映射表中的模型直接显示原始名称。

```typescript
const MODEL_DISPLAY_NAMES: Record<string, Record<string, [string, string]>> = {
  deepseek: {
    "deepseek-chat":     ["DeepSeek V3", "快速、经济"],
    "deepseek-reasoner": ["DeepSeek R1", "深度推理"],
  },
  zhipu: {
    "glm-4-plus":  ["GLM-4 Plus", "旗舰模型"],
    "glm-4-flash": ["GLM-4 Flash", "快速（含免费额度）"],
  },
  dashscope: {
    "qwen-max":   ["通义千问 Max", "旗舰模型"],
    "qwen-plus":  ["通义千问 Plus", "推荐"],
    "qwen-turbo": ["通义千问 Turbo", "快速、经济"],
  },
};
```

**理由**：供应商随时会新增/下线模型，硬编码列表会导致 UI 与现实脱节。映射表作为渐进增强层，不影响基本功能。

---

### D5：首次使用软提示（Option B）

**选择**：App 启动无需配置即可打开，发送第一条消息时判断状态：
- 试用未激活且未配置密钥 → 激活试用并提示"试用已开始，剩余 14 天"
- 试用已过期且未配置密钥 → 阻断，跳转配置面板
- 已配置密钥 → 正常使用

**理由**：让用户先感受产品价值再要求操作，比强制引导向导有更高转化率。

---

### D6：移除路由策略和优先级 Tab

**选择**：完全从 UI 删除，不保留"高级模式"入口。对应后端端点（`POST /api/models/routing`、`POST /api/models/priorities`）保留但不再从前端调用（为未来可能恢复保留后端能力）。

**理由**：用途路由对 4 供应商单激活模式无意义；保留"高级设置"折叠会让科研用户感到不安（"我需要配置这个吗？"）。

## Risks / Trade-offs

| 风险 | 缓解措施 |
|---|---|
| 内嵌试用密钥被提取/滥用 | 在 DeepSeek 后台设置速率限制；发现滥用后轮换密钥（更新发布版本） |
| 供应商动态模型 API 偶尔失败 | 前端显示"模型获取失败，请检查密钥或网络"，并提供手动输入模型名的降级入口 |
| 廉价模型偏好列表与供应商现有模型不匹配 | 降级策略：如偏好列表全部不可用，则使用主模型（用户已选择的那个） |
| 删除路由功能可能影响当前高级用户 | 现有多供应商配置数据在迁移时保留，但仅第一个激活的供应商生效；路由端点后端保留 |
| 14 天试用被清数据重置 | 早期阶段可接受，后续可叠加设备指纹方案 |

## Migration Plan

1. **数据库迁移**：`app_settings` 表新增 `trial_install_date` / `trial_activated` 字段（可 NULL，首次发消息时写入）
2. **现有模型配置**：`model_configs` 表不破坏，但 UI 仅展示 4 个支持的供应商；已配置其他供应商的用户需重新选择
3. **路由配置数据**：`preferred_provider_by_purpose` 数据保留在数据库，后端不再从前端写入，但存量数据无害
4. **前端部署**：RoutingTab / PriorityTab 组件文件删除；ModelConfigPanel 重写后旧状态自动失效

**回滚策略**：git revert PR，前端静态文件重新部署，数据库新增字段可保留（不影响旧版本）。
