## Context

当前 Nini 工具层同时承担三种职责：原子能力、复合工作流、对前端和会话状态的辅助操作。这些职责统一以 Function Tool 的形式暴露给模型，导致两个直接后果：

- 工具选择空间过大，模型需要在 30 个并列工具之间做选择，提示词噪声较高。
- 高自由度工具 `run_code` 承担了大量不该由自由脚本承担的状态管理工作，例如中间数据集命名、图表路径管理、失败后的修复策略。

从实际会话 `13839f39e762` 看，失败并非来自统计能力不足，而是来自状态不稳定：

- 图表写入路径不存在，导致整个绘图脚本失败。
- 合并后的中间数据集没有显式持久化，后续步骤反复报“数据集不存在”。
- 单次失败后只能重新生成大段代码，而不是对失败位置做局部修补。

仓库当前已具备若干可复用基础设施：`Session`、`WorkspaceManager`、`ArtifactStorage`、统一 `SkillResult`、工作区执行历史与图表/报告产物链路。本次设计不引入新的外部依赖，重点是在现有后端骨架上重新组织工具边界和执行契约。

## Goals / Non-Goals

**Goals:**

- 将模型可见工具收敛为少量基础工具，降低选择成本和提示词负担。
- 为数据处理、代码执行、图表、报告建立显式资源生命周期，消除“最近一次结果”与自由路径写入的隐式状态。
- 为代码和数据处理提供局部 patch 与失败恢复能力，避免一次错误后整段重生成。
- 将复合分析能力改为可观测的编排层，复用基础工具，减少重复逻辑。
- 让工作区、报告、图表、脚本与中间数据集都能通过统一资源标识追踪。

**Non-Goals:**

- 不在本次变更中引入新的统计方法或新的科研分析领域功能。
- 不追求保留 30 个旧工具的向后兼容别名。
- 不在本次变更中重做前端整体交互，只做必要的资源与事件适配。
- 不把所有数据操作都退化为通用代码执行；结构化高频任务优先走基础工具。

## Decisions

### 决策 1：采用“基础工具层 + 内部编排层”替代扁平 30 工具暴露

决定：

- 对模型仅暴露 9 个基础工具：任务状态、数据目录、数据变换、统计检验、统计建模、统计解读、图表会话、报告会话、工作区会话。
- 将完整比较、完整 ANOVA、相关分析、回归分析等高层能力保留为内部编排层，不再作为与基础工具同级的模型接口。

原因：

- 会话证据表明，模型高频用的是少数基础能力，复合能力更适合作为编排模板而不是公开接口。
- 分层后可以把“工具选择”问题转为“流程编排”问题，降低模型误选概率。

备选方案：

- 保留 30 工具，仅通过 `expose_to_llm=False` 隐藏一部分。
  放弃原因：只能减少可见数量，无法解决隐式状态和失败恢复问题。

### 决策 2：新增统一资源标识，替代纯文本名称和“latest_*”隐式引用

决定：

- 为数据集、图表规格、报告、脚本、统计结果、工作区文件引入统一 `resource_id`。
- 图表、报告、脚本等后续操作优先使用 `resource_id`，名称仅用于展示。

原因：

- 纯文本名称容易重复、漂移或被模型重写。
- 当前 `latest_chart` 之类的隐式状态在长会话和失败恢复场景下不稳定。

备选方案：

- 继续使用名称查找并通过提示词强调命名规范。
  放弃原因：对模型约束弱，无法消除状态漂移。

### 决策 3：将 `run_code` / `run_r_code` 重构为持久化脚本会话

决定：

- 统一为 `code_session` 概念，支持 `create_script`、`run_script`、`patch_script`、`rerun`、`promote_output`。
- Python/R 仅作为 `language` 参数差异，生命周期与结果契约一致。

原因：

- `edit_file` 的核心优势是“局部修改而非整段重写”；代码执行场景需要同类能力。
- 失败恢复必须绑定到脚本实体，否则模型只能重新生成完整代码。

备选方案：

- 保持 `run_code`，仅补充更多 `save_as` / `persist_df` 参数。
  放弃原因：仍是一次性执行模型，无法表达 patch 和增量重跑。

### 决策 4：将高频数据处理从自由代码迁移到结构化数据流水线

决定：

- 新增 `dataset_transform`，采用步骤数组执行常见清洗、拼接、聚合、衍生列、去重、过滤。
- 只保留有限枚举操作；复杂非标准处理仍允许回退到 `code_session`。

原因：

- 数据清洗、数据汇总这类任务高度结构化，没必要每次都重新生成脚本。
- 结构化步骤天然适合失败定位、局部 patch、重放和审计。

备选方案：

- 所有数据处理继续交给脚本执行。
  放弃原因：灵活但不可控，失败粒度过大。

### 决策 5：图表和报告改为会话资源，而不是一次性输出

决定：

- `chart_session` 先创建和更新图表规格，再执行导出。
- `report_session` 以章节块管理报告，支持 `patch_section`、附件绑定、导出。

原因：

- 现有“生成一次图表/报告，再靠文件修改或 latest 引用继续操作”的方式不稳定。
- 图表与报告天然是可继续编辑的会话对象，不是一次性副产物。

备选方案：

- 保持 `create_chart` / `generate_report` 一次性输出，再用 `edit_file` 修补。
  放弃原因：文本文件层面可以修补，但语义层面缺少稳定对象和结构。

### 决策 6：不保留旧工具名兼容层

决定：

- 旧 30 个工具从模型正式暴露接口中移除，不提供别名转发。
- 前端和测试直接迁移到新的基础工具与编排层。

原因：

- 兼容层会迫使新旧执行模型同时存在，拉长迁移周期并增加行为分叉。
- 本次目标是彻底收敛，不是包一层名称别名。

备选方案：

- 保留旧工具名，内部转发到新实现。
  放弃原因：短期轻松，但长期难以清理，提示词也更难收敛。

## Risks / Trade-offs

- [风险：一次性替换 30 工具导致回归面大] → 先稳定基础类型与工具注册，再迁移编排层和前端调用，使用会话回放样例做端到端回归。
- [风险：`dataset_transform` schema 过宽，反而形成新的大黑盒] → 仅支持明确枚举的高频操作，复杂逻辑强制走 `code_session`。
- [风险：脚本会话持久化后工作区和执行历史数据量增长] → 对脚本、运行记录和中间产物设置目录结构与清理策略，避免无界增长。
- [风险：前端仍依赖旧事件/旧工具名] → 在 `runner` 与 WebSocket 事件中补充新资源类型和工具名映射测试。
- [风险：报告/图表会话引入更多资源 ID，增加模型负担] → 所有基础工具返回简洁的统一资源摘要，并在提示词中明确“继续操作优先使用上一步返回的 resource_id”。

## Migration Plan

1. 先引入统一资源类型、基础工具骨架和脚本会话目录结构，不立即删除旧实现。
2. 将注册表切换为仅暴露新基础工具，同时把复合分析改为内部编排调用新工具。
3. 迁移报告、图表、工作区和执行历史链路到新资源模型。
4. 更新前端与测试，确保工作区面板、执行历史、报告下载、图表预览均识别新资源。
5. 删除旧工具注册与不再使用的隐式状态代码。

回滚策略：

- 若新工具链在回归中出现阻塞，可临时恢复旧注册表分支；由于本次不保留正式兼容层，回滚以代码级整体回滚为主，而不是运行时开关。

## Open Questions

- 是否需要在第一版就为 `dataset_transform` 提供“查看执行计划差异”能力，还是先只支持 patch 单一步骤。
- 脚本会话的历史保留上限应放在工作区层还是单独配置项中。
- 前端是否在第一阶段直接展示 `resource_id` 关联关系，还是先仅用于后端和调试信息。
