## Context

`add-recipe-center-mvp` 解决的是“如何把任务跑起来”，但科研场景真正决定可用性的，是产出能否被复查。当前 Nini 已有引用标注、知识检索、报告会话和工作区资源，但这些能力更多是“展示层引用”而不是“结论级证据绑定”，也缺少统一的 METHODS 记录。

本设计将证据可追溯拆成 B1/B2 两个 change。B1 只做结论与来源的绑定、最小 Evidence Block 和 METHODS 台账；B2 再做冲突检测、置信评分与无证据结论拦截。这样可以避免在证据数据模型尚未稳定时过早引入复杂校验流水线。

## Goals / Non-Goals

**Goals:**
- 为关键结论建立稳定 `claim_id` 与来源记录的映射。
- 统一不同来源的最小溯源字段，让报告与引用展示复用同一份证据数据。
- 生成最小可用的 Evidence Block 与 METHODS v1 内容。
- 保持与现有报告会话、引用展示和工作区资源兼容。

**Non-Goals:**
- 不在本 change 中实现冲突检测、置信度评分与结论真实性判定。
- 不在本 change 中阻止无证据结论进入最终摘要。
- 不在本 change 中实现 Word/PPTX/LaTeX 导出模板。

## Decisions

### 决策 1：先引入 `claim_id + source_record`，不直接做图数据库

证据绑定的核心是能稳定回溯，而不是一开始就上复杂图结构。MVP 阶段只要求结论有唯一 `claim_id`，来源有统一 `source_record`，两者可在报告和回答中双向映射。

备选方案：
- 方案 A：直接落地完整 Citation Graph/图数据库式结构。
- 方案 B：先定义可序列化的 `claim_id + source_record + evidence_block` 模型。

选择 B，因为现阶段重点是产品可追溯性和实现成本控制，而不是复杂查询能力。

### 决策 2：METHODS 台账作为独立资源层，而不是临时拼接文本

如果只在最终导出时临时拼接 METHODS 文本，就无法支持报告会话增量更新，也无法让后续 Claim 校验复用执行上下文。METHODS 必须先是结构化台账，再派生文本。

备选方案：
- 方案 A：导出时临时根据日志拼文本。
- 方案 B：每个关键步骤写入结构化 METHODS 台账，再生成 METHODS v1。

选择 B，因为它能支持报告增量更新和后续评测复用。

### 决策 3：复用现有引用与报告能力，不新增独立证据查看面板

本 change 优先让回答和报告可追溯，不新增完整证据审查 UI。引用详情、报告 Evidence Block 与报告 METHODS 区块足以满足 B1 目标。

备选方案：
- 方案 A：新增独立 Evidence Inspector 页面。
- 方案 B：先扩展引用详情和报告渲染。

选择 B，因为新增独立页面会提前把 Phase C/D 的产品化工作卷入当前 change。

## Risks / Trade-offs

- [来源字段不一致] -> 先定义跨来源的最小公共字段，其余作为扩展属性保留。
- [METHODS 记录不完整] -> 明确允许输出部分内容并标记缺失字段，避免伪造完整方法说明。
- [报告负载变大] -> 只对关键结论生成 Evidence Block，不强制所有句子都携带证据块。
- [引用与报告模型分叉] -> 强制两者复用同一来源记录与 `claim_id`，避免双份数据结构。

## Migration Plan

1. 先在后端定义来源归一化结构、`claim_id` 映射与 METHODS 台账写入点。
2. 扩展报告资源结构，使其能够持久化 Evidence Block 与 METHODS v1。
3. 扩展引用详情展示与报告渲染，但默认保持现有引用入口不变。
4. 如出现性能或渲染回退，可先关闭 Evidence Block 展示，仅保留后台证据数据写入。

## Open Questions

- `claim_id` 的生成粒度是以最终句子、段落还是报告章节为主。
- 知识检索之外的网页抓取或工作区资源，是否需要统一原始快照保留策略。
- METHODS 台账与现有执行日志之间由谁作为权威来源。
