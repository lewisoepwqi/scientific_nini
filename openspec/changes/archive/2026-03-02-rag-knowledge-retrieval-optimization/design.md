## Context

Nini 项目的知识检索系统当前采用文档级索引，将整个 Markdown 文件作为检索单元。这种粗粒度索引在科研数据分析场景中存在明显局限：用户查询特定统计方法的参数说明时，系统只能返回整个文档，无法精确定位到相关段落。

基于 PageIndex 技术调研，我们决定引入层次化索引架构（文档/章节/段落三级），结合查询意图路由和多阶段检索流程，提升检索精准度和用户体验。

## Goals / Non-Goals

**Goals:**
- 实现 L0/L1/L2 三级层次化索引，支持 Markdown 文档结构化解析
- 实现查询意图分类（概念/方法/参考/代码/比较/故障排查）和智能路由
- 实现多路召回（BM25 + 向量 + 长期记忆）和 RRF 融合
- 实现轻量级 Cross-Encoder 重排序，提升 Top-K 相关性
- 提供统一检索接口，向后兼容现有 `KnowledgeLoader`

**Non-Goals:**
- 不替换现有的向量存储实现（基于 LlamaIndex）
- 不修改知识 Markdown 文件的格式要求
- 不引入外部向量数据库（保持本地优先）
- 不实现 PageIndex 的无向量推理导航（保留向量检索作为基础）

## Decisions

### 1. 索引架构：独立三级索引 vs 嵌套树结构

**决策**：采用独立三级索引 + 关系映射表，而非完全嵌套的树对象

**理由**：
- 独立索引允许并行构建和检索，性能更好
- 关系映射表支持灵活的双向导航
- 更容易实现增量更新

**替代方案**：完全嵌套的树对象（类似 PageIndex 的 JSON 结构）
- 拒绝原因：Python 中嵌套对象操作开销大，不利于并行处理

### 2. Markdown 解析：正则 vs 专用库

**决策**：使用正则表达式 + 简单状态机解析 Markdown 结构

**理由**：
- 知识文件格式相对规范，不需要完整 Markdown 解析
- 避免引入额外依赖（如 `markdown-it-py`）
- 更好的中文标题支持

**实现要点**：
- 识别 `# ## ###` 标题层级
- 提取标题文本和位置信息
- 按段落边界进行语义分块

### 3. 查询意图分类：规则引擎 vs 轻量级模型

**决策**：采用规则引擎 + 关键词匹配，而非训练分类模型

**理由**：
- 科研查询意图类别明确，规则覆盖率高
- 零额外依赖，推理速度快（<1ms）
- 易于维护和调试

**规则示例**：
- 概念查询：`什么是.*`、`.*是什么`、`解释.*`
- 方法查询：`如何.*`、`怎么.*`、`步骤`
- 参考查询：`参数`、`返回值`、`说明`

### 4. 检索策略：层次路由 vs 全层搜索

**决策**：根据意图类型路由到主要层级，辅以跨层搜索

**路由映射**：
| 意图类型 | 主要层级 | Top-K |
|----------|----------|-------|
| concept | L0 (文档) | 3 |
| how-to | L1 (章节) | 5 |
| reference/code | L2 (段落) | 5 |
| comparison | L0 + L1 | 4 |
| troubleshoot | L1 + L2 | 5 |

**理由**：
- 减少无效检索，提升效率
- 不同意图类型的信息密度需求不同

### 5. 重排序模型：本地 Cross-Encoder vs API

**决策**：使用本地轻量级 Cross-Encoder（BGE-Reranker）

**理由**：
- 保持本地优先设计原则
- 单次重排序延迟可接受（~50-100ms）
- 模型大小适中（~1GB）

**回退策略**：模型加载失败时跳过重排序

### 6. 索引持久化：JSON vs 专用格式

**决策**：使用 JSON + pickle 混合格式

**结构**：
```
data/hierarchical_index/
├── l0_index.json      # 文档级索引（元信息）
├── l1_index.json      # 章节级索引
├── l2_vectors.pkl     # 段落级向量（pickle 压缩）
├── parent_map.json    # 父子关系映射
└── version.json       # 版本和哈希信息
```

**理由**：
- JSON 便于调试和版本控制
- Pickle 适合存储密集向量数据
- 分离存储支持独立加载

## Risks / Trade-offs

| Risk | 影响 | 缓解措施 |
|------|------|----------|
| 索引构建时间过长 | 中 | 后台异步构建 + 增量更新 |
| 内存占用增加 | 中 | 索引按需加载 + LRU 缓存 |
| Cross-Encoder 推理延迟 | 低 | 异步重排序 + 可配置关闭 |
| 意图分类误判 | 低 | 混合路由 + 用户反馈调优 |
| 向后兼容性问题 | 低 | 保留 KnowledgeLoader 接口 |

**Trade-offs**：
- 存储空间换检索速度：三级索引占用更多磁盘空间，但检索更快
- 复杂度换精准度：更复杂的架构带来维护成本，但检索质量显著提升

## Migration Plan

### Phase 1: 部署前准备
1. 创建新的索引目录 `data/hierarchical_index/`
2. 配置开关 `settings.enable_hierarchical_index`（默认 False）
3. 实现向后兼容层

### Phase 2: 灰度切换
1. 启用新索引构建（后台）
2. 通过配置切换部分流量到新接口
3. 监控检索质量和延迟

### Phase 3: 全面切换
1. 默认启用新检索接口
2. 保留旧接口作为回退
3. 清理旧索引数据

### Rollback Strategy
- 配置切换 `settings.knowledge_strategy = "bm25"` 立即回退到旧实现
- 新旧索引独立存储，互不影响
- 代码层面保留 `KnowledgeLoader` 类作为适配器

## Open Questions

2. **缓存策略**：查询结果缓存的 TTL 设置为多少合适？
3. **多语言支持**：当前主要面向中文，是否需要支持英文混合查询？
4. **评估指标**：如何量化检索质量提升？是否需要构建评估数据集？
