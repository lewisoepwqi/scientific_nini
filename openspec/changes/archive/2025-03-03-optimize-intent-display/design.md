## Context

当前 Nini 前端使用三个主要组件展示系统理解：

1. **IntentSummaryCard** (`ChatPanel.tsx:73`)：位于输入框上方，展示候选能力、推荐工具、激活技能、澄清建议。组件代码 223 行，一次展示大量技术信息。

2. **IntentTimelineItem** (`ChatPanel.tsx:111`)：在用户消息后展示意图理解确认，与 SummaryCard 内容有重叠。

3. **KnowledgeRetrievalView** + MessageBubble 内联展示：检索结果以相关性分数形式直接插入消息内容中。

### 当前问题
- 信息密度过高：普通用户面对"候选能力"、"显式技能调用"、"规则版 v2"等技术术语感到困惑
- 视觉干扰：IntentSummaryCard 固定在输入框上方，挤压对话空间
- 阅读中断：知识检索结果内联展示，破坏回答的连贯性
- 重复信息：SummaryCard 和 TimelineItem 展示相似内容

## Goals / Non-Goals

**Goals:**
- 降低系统理解展示的认知负荷，让普通用户更容易理解
- 采用渐进式披露设计，默认简化、按需展开详情
- 重构知识引用展示，采用学术界熟悉的引用标注模式
- 提供用户控制选项，满足不同专业级别用户的需求
- 保持现有功能完整，仅优化展示方式

**Non-Goals:**
- 不修改意图分析的后端逻辑（`src/nini/intent/`）
- 不修改知识检索的后端逻辑（`src/nini/knowledge/`）
- 不修改 WebSocket 事件协议
- 不引入新的 AI 模型或算法

## Decisions

### 决策 1：IntentSummaryCard 采用折叠式设计
**选择**：默认只展示一句话概括 + "查看详情"入口
**替代方案**：完全移除 SummaryCard，只在 TimelineItem 中展示
**理由**：
- SummaryCard 在输入阶段的预判价值仍然存在，帮助用户确认表达清晰
- 完全移除会失去预判能力，用户只能在发送后才能确认理解是否正确
- 折叠式设计平衡了简洁性和功能性

### 决策 2：使用引用标注 [1], [2] 替代内联检索展示
**选择**：在 AI 回答中标注 [1], [2]，底部统一列出参考来源
**替代方案**：侧边栏展示检索结果、悬浮卡片展示
**理由**：
- 引用标注是学术界标准做法，科研用户熟悉此模式
- 不占用额外屏幕空间，保持回答区域简洁
- 悬停查看详情提供渐进式信息获取
- 侧边栏方案会挤压主要内容区域

### 决策 3：持久化用户展示偏好到 localStorage
**选择**：前端 localStorage 存储用户偏好
**替代方案**：后端用户配置存储、每次会话重置
**理由**：
- 展示偏好是纯前端体验问题，无需后端参与
- localStorage 简单可靠，无需 API 调用
- 用户期望偏好跨会话保持

### 决策 4：使用可信度标签替代数字分数
**选择**："高可信度"/"一般参考"标签
**替代方案**：保留数字分数、使用星级评分
**理由**：
- 普通用户不理解 0.85 意味着什么
- 标签形式更直观，用户能快速判断可信度
- 星级评分可能过于细化，增加认知负担

### 决策 5：默认简化模式，可切换到详细模式
**选择**：默认简化，专业用户手动切换到详细
**替代方案**：默认详细、根据用户行为自动判断
**理由**：
- 大多数用户是普通科研人员，不需要技术细节
- 自动判断复杂且可能出错，显式控制更可靠
- 简化模式更符合"不要让我思考"原则

## Risks / Trade-offs

### [Risk] 老用户习惯现有展示方式，变更可能引起不适
**Mitigation**：
- 保留"详细模式"选项，老用户可以一键切换回熟悉界面
- 变更首次访问时展示引导提示
- 监控用户反馈，必要时调整默认设置

### [Risk] 简化展示可能让专业用户感到信息不足
**Mitigation**：
- 详细模式保留所有技术信息
- 在设置中明显标注详细模式入口
- 考虑根据用户行为智能推荐模式（后续迭代）

### [Risk] 引用标注与现有 Markdown 解析冲突
**Mitigation**：
- 使用正则表达式 `\[(\d+)\]` 识别引用标注
- 确保标注在 Markdown 链接 `[]()` 之前解析
- 充分测试边界情况

### [Risk] localStorage 可能因隐私设置被禁用
**Mitigation**：
- 使用 try-catch 包装 localStorage 操作
- 降级到内存存储，每次会话重置为可接受行为
- 不依赖 localStorage 进行核心功能

## Migration Plan

本次变更为纯前端展示层优化，无数据迁移需求。

**部署步骤**：
1. 开发阶段使用 feature flag 控制新组件
2. 内部测试验证所有三种模式（简化/详细/隐藏）
3. 灰度发布，监控用户反馈
4. 全量发布

**回滚策略**：
- 纯前端变更，回滚只需重新部署上一版本
- 无数据状态需要恢复
