## Context

`add-logging-foundation` 解决的是日志底座问题，但底座建立后，当前运行期日志仍然存在三个明显短板：

1. 关键失败路径中仍有不少 `logger.error()` 不带 traceback，排障信息不足。
2. 代码中存在较多 `except Exception: pass`，但它们并不都同等重要，需要按风险分层处理。
3. 模型调用、工具执行、检索、沙箱执行和 Agent 单轮执行缺少统一耗时信号，性能定位依然困难。

这次 change 跨越 `agent/`、`api/`、`tools/`、`knowledge/`、`sandbox/` 和 `workspace/`，属于典型的横切可观测性增强，需要先约束范围和日志语义，避免演变成全仓重写。

## Goals / Non-Goals

**Goals:**

- 为关键意外失败路径补齐可诊断的错误日志
- 按风险分层治理静默吞错，优先清理业务失败路径
- 为关键执行链路补充统一耗时日志
- 为当前明显薄弱的模块补齐基础操作日志
- 保持新增日志语义可测试、可验证

**Non-Goals:**

- 不要求本 change 清零所有 `except Exception: pass`
- 不把所有 warning 或降级分支升级成 error
- 不引入日志采样、告警平台或日志查看 CLI
- 不要求把所有日志改成 JSON 结构化输出
- 不改变 WebSocket 或 HTTP 对外协议

## Decisions

### 决策 1：异常增强只覆盖“意外失败路径”

本 change 中应优先为这些路径补 `exc_info=True`：

- 捕获到未预期异常并继续向上返回失败结果
- 导致当前任务、工具调用、消息处理或执行链中断的异常
- 会明显影响排障效率的关键边界点

不强制增强的情况：

- 明确预期的业务拒绝、降级或兼容分支
- 缺少 traceback 也不会影响问题定位的用户输入错误

原因：

- 不是所有异常都需要完整堆栈
- 如果对所有路径都打印 traceback，会显著增加噪声

### 决策 2：静默吞错按风险分层治理

处理原则：

- 业务失败路径中的静默吞错应改为 warning/error 日志
- 清理、关闭、回收路径中的静默吞错可以保守保留，但至少记录 debug
- 不要求一次性处理所有 39 处

原因：

- 当前静默吞错分布在多个模块，风险不均匀
- 一次性全量清理容易引入行为变化和过宽变更面

备选方案：

- 方案 A：一次性消灭全部 `except Exception: pass`
  - 缺点：范围失控，收益与风险不匹配
- 方案 B：维持现状，仅靠人工排查
  - 缺点：问题继续黑箱化

结论：采用“业务优先、清理次之”的分层治理。

### 决策 3：耗时日志只覆盖关键链路，字段命名尽量统一

首批耗时点：

- Agent 单轮执行
- 模型调用
- 工具执行
- 检索
- 沙箱执行

建议统一使用 `duration_ms` 语义，并在文本日志中保持尽量一致的表达方式。

原因：

- 关键链路足以支撑大部分性能定位
- 统一字段语义有利于后续转向 JSON formatter 或结构化日志
- 避免在低价值路径增加日志噪声

### 决策 4：薄弱模块先补基础日志，而不是补“全覆盖日志”

优先补充的区域：

- `workspace/manager.py`
- `api/routes.py` 中关键文件与会话操作
- `agent/model_resolver.py`

补充内容控制在：

- 开始
- 失败
- 成功/完成

原因：

- 这些模块当前不是完全无日志，就是关键路径日志过薄
- 先补基础操作日志比追求“每个分支都打日志”更可控

### 决策 5：测试关注“日志是否存在且语义正确”，不绑定具体文本细节

测试重点：

- 关键异常路径是否携带 traceback
- 关键耗时日志是否输出
- 原本静默的高风险路径是否有日志

不建议让测试强绑定完整日志文案，否则会放大文案调整成本。

## Risks / Trade-offs

- [日志噪声增加] → 新增 traceback 和耗时日志可能抬高日志量；通过只覆盖关键路径缓解。
- [边界判断不一致] → 哪些异常算“意外失败”可能因模块而异；通过优先清单和测试覆盖缓解。
- [行为变更风险] → 修改静默吞错时可能暴露旧问题；通过先改记录、不改主流程控制来缓解。
- [性能开销] → 耗时测量和新增日志会引入少量开销；通过只覆盖关键链路、避免过密打点缓解。

## Migration Plan

1. 梳理关键异常路径和高风险静默吞错点，确定首批治理名单。
2. 先在 `runner.py`、`model_resolver.py`、`websocket.py`、`tools/registry.py` 等关键模块补 traceback。
3. 再为模型调用、工具执行、检索、沙箱和 Agent 单轮执行增加耗时日志。
4. 最后为 `workspace/manager.py` 和关键 API 操作补基础日志。
5. 增加测试或现有测试中的日志断言，验证新增日志语义。

回滚方式：

- 若某类新增日志带来过量噪声，可回退对应模块的日志增强，而不影响日志底座
- 若某个静默吞错修复暴露出兼容性问题，可先降级为 debug 记录，保留原执行路径
