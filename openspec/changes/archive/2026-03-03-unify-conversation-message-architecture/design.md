## Context

当前对话系统同时维护三套相互不完全一致的表示：

- WebSocket 实时事件：已经引入 `message_id`、`turn_id`、`operation`
- Session / ConversationMemory 持久化：多数消息仅保存 `role/content`，缺少稳定身份
- 前端历史恢复：通过 `buildMessagesFromHistory()` 从历史记录二次推导展示消息

这种结构让“实时显示正确”和“刷新后恢复正确”变成两个独立问题。前一次修复已经证明，即便实时链路引入了去重协议，只要历史 API、Session 落盘和前端恢复没有使用同一消息模型，重复、错序、错误合并仍会反复出现。

同时，仓库还存在两类架构漂移：

- 后端重复的消息历史路由，返回契约并不一致
- 前端主 store 与未接线 slice 并存，容易导致修复落在无效实现

这次设计的目标不是继续修一个重复 bug，而是把“对话消息”收敛为一个可持久化、可恢复、可对账的系统模型。

## Goals / Non-Goals

**Goals:**

- 定义统一的消息生命周期模型，覆盖助手文本、推理、工具调用结果及其流式更新
- 让 `message_id`、`turn_id`、消息类型和必要元数据进入 Session 持久化与历史 API
- 让前端实时消费和历史恢复走同一套归并规则，避免“双模型”
- 为刷新、重连、重试、中断提供明确的对账策略
- 收敛重复的历史路由与前端 store 入口，减少后续维护偏差

**Non-Goals:**

- 不在本次变更中重做聊天 UI 视觉样式
- 不引入新的外部消息总线、数据库或第三方状态同步组件
- 不重写全部会话存储格式；仅在现有 JSONL/本地文件体系上扩展必要字段并提供兼容迁移
- 不处理与消息架构无关的图表性能、成本面板或通用前端拆分问题

## Decisions

### 决策 1：建立后端规范消息记录（canonical message record）

采用统一的消息记录结构，至少包含：

- `message_id`
- `turn_id`
- `role`
- `content`
- `event_type`
- `operation`
- `tool_call_id` / `reasoning_id` 等关联字段
- `_ts`

理由：

- 当前 `Session.add_message()` 与 `add_reasoning()` 的写法不一致，导致不同类型消息进入历史后丢失关键身份字段
- 只有让身份随消息一起落盘，刷新和重连后才能继续做幂等归并

备选方案：

- 仅在 WebSocket 层维护 `message_id`，历史恢复继续靠内容推断
  - 放弃，已证明无法覆盖刷新恢复和断线场景
- 另建独立 event log，不改现有 Session message
  - 放弃，迁移成本更高，会在维护期引入第二套持久化体系

### 决策 2：统一历史读取契约，只保留一个消息历史 API

`/api/sessions/{id}/messages` 将成为唯一历史消息读取接口，返回完整消息元数据。重复路由要么删除，要么降级为内部适配层，但不能继续作为对外契约存在。

理由：

- 同一路径存在两套返回结构会导致前端和测试对“真实契约”形成错误认知
- 统一接口后，实时与恢复可以对齐到同一 message schema

备选方案：

- 保留双路由，通过注册顺序控制生效实现
  - 放弃，这会持续制造维护陷阱

### 决策 3：前端只保留一个消息归并入口

实时 WebSocket 事件与历史加载结果都先进入统一 normalizer，再落到 `messages` 展示状态。`_messageBuffer` 只负责 in-flight 消息状态，不承担历史推导责任。

理由：

- 当前 `handleEvent()` 和 `buildMessagesFromHistory()` 在做相似但不等价的事情
- 归并规则分散会导致同一问题修两遍且仍不一致

备选方案：

- 保持一套实时 reducer、一套历史恢复 builder
  - 放弃，根本缺陷就是双入口归并

### 决策 4：把重连视为“对账”而不是“继续流”

WebSocket 重连后，客户端不假设可以无缝续接流式 token，而是执行一次 session reconcile：

- 清理本地未完成 buffer
- 拉取当前 session 历史
- 基于 `message_id` / `turn_id` 合并或替换现有展示状态

理由：

- 当前后端没有 `last_event_id` 回放机制，假装支持续流会制造更多幽灵状态
- 在现有架构下，拉历史对账是成本最低且最稳妥的恢复策略

备选方案：

- 新增事件序列回放协议
  - 暂不采用，可作为后续增强；当前维护阶段优先收敛正确性

### 决策 5：分阶段迁移，先兼容旧历史，再逐步收紧

实现顺序采用“先读兼容、后写规范、再删旧路”的方式：

- 读取历史时兼容缺少 `message_id` 的旧消息
- 新写入消息必须带完整元数据
- 当历史 API 与前端完成切换后，移除重复路由和无效 store 实现

理由：

- 仓库中已有大量旧会话数据，不能一次性要求全部历史记录满足新格式
- 维护期需要优先保证老数据可读、新数据正确

备选方案：

- 一次性升级所有历史文件
  - 放弃，风险高且回滚困难

## Risks / Trade-offs

- [旧历史记录缺少身份字段] → 读取阶段提供兼容归并策略，并把兼容逻辑限定在 history normalizer 内
- [前后端同时修改消息模型，回归面较大] → 先补契约测试，再按“后端 schema → API → 前端 normalizer”顺序推进
- [删除重复路由或未接线 store 时可能误删隐性依赖] → 先通过搜索与测试确认调用路径，再分提交清理
- [重连改为对账后，用户会看到一次状态重刷] → 接受该权衡，优先保证一致性而非伪连续流体验
- [JSONL 历史体积变大] → 仅新增必要标识字段，继续保留大 payload 引用化策略

## Migration Plan

1. 扩展后端 Session / ConversationMemory 写入结构，使新消息写入完整元数据
2. 统一历史消息 API 返回格式，并为旧记录提供兼容字段填充
3. 在前端引入统一消息 normalizer，让历史加载和实时事件共享归并逻辑
4. 接入重连后的 session reconcile 流程，处理 buffer 清理与状态重建
5. 删除或封存重复历史路由与未接线 store 实现
6. 通过回归测试验证：正常流式、工具替换、刷新恢复、断线重连、重试上一轮

回滚策略：

- 若前端归并逻辑异常，可暂时回滚到旧 history builder，但保留后端新增字段不影响兼容
- 若统一历史接口出现问题，可短期恢复旧实现，但必须保留返回字段超集，避免再次分叉
