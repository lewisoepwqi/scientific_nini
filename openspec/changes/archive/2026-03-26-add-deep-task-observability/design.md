## Context

当前仓库已经有 harness trace、completion check、blocked 事件和部分成本透明能力，但这些能力还没有以 deep task 为中心收束起来。进入维护阶段后，真正影响迭代速度的不是缺少新功能，而是遇到失败时定位慢、发布前没有稳定的 Recipe 回归门禁、以及成本或重试策略分散在各处。

这个 change 的目标是建立“最小运维闭环”：每个 deep task 能被唯一识别，失败能被回放定位，发布前能被基准集拦截，超预算时能被感知。

## Goals / Non-Goals

**Goals:**
- 为 deep task 建立端到端 `task_id` 与关键指标采集。
- 为外部依赖调用补齐重试、幂等与超时分级策略。
- 为核心 Recipe 建立发布前回放基准和门禁阈值。
- 为预算超限提供统一告警信号。

**Non-Goals:**
- 不建设独立的观测后台或接入外部遥测 SaaS。
- 不在本 change 中新增业务能力或调整证据/导出逻辑。
- 不覆盖所有历史会话，只要求对新的 deep task 运行生效。
- 不要求在本阶段新增独立预算告警 UI，预算超限以 trace、日志和诊断事件可见为主。

## Decisions

### 决策 1：以 deep task 为主键，而不是以会话或单次工具调用为中心

会话维度太粗，工具调用维度又太细。维护阶段真正需要的是“这一类深任务从开始到交付经历了什么”，因此使用 `task_id` 作为主键更适合串联入口、运行、产物和回放。

备选方案：
- 方案 A：继续以会话 ID 为主要关联键。
- 方案 B：为 deep task 引入独立 `task_id`，并在相关事件中贯穿。

选择 B，因为 deep task 已经成为规划、产物和预算的核心边界。

### 决策 2：可靠性策略只对外部依赖和关键副作用做幂等保护

不是所有步骤都值得做复杂幂等控制。MVP 阶段重点保护模型调用、检索、导出和正式产物登记等关键副作用。

备选方案：
- 方案 A：为所有动作设计统一幂等层。
- 方案 B：先覆盖外部依赖与关键副作用。

选择 B，因为它能在可控成本下先解决最痛的问题。

### 决策 3：回归门禁以核心 Recipe 为中心

维护阶段最有价值的不是大而全评测，而是每次发布前能快速知道高频 deep task 是否退化。因此回放与门禁优先围绕核心 Recipe 建立。

备选方案：
- 方案 A：先做全量开放式评测。
- 方案 B：先做核心 Recipe 基准集与阈值门禁。

选择 B，因为它更贴近当前迭代目标，也更容易纳入发布流程。

## Risks / Trade-offs

- [标识链路接入面广] -> 先覆盖 deep task 关键事件与正式产物，不追求一次性覆盖所有内部日志。
- [重试导致重复副作用] -> 对正式产物登记和导出作业优先增加幂等保护。
- [门禁过严拖慢发布] -> 先用核心 Recipe 阈值，允许灰度观察，再逐步提高要求。
- [预算阈值误报] -> 先记录并告警，避免在第一阶段直接强制中断任务。

## Migration Plan

1. 为新的 deep task 运行分配并透传 `task_id`。
2. 在 harness runtime 中接入分级重试、幂等保护与超时策略。
3. 建立核心 Recipe 基准集与发布前回放脚本。
4. 接入预算告警与关键事件字段；如出现噪音，可先只记录不展示。

## Rollout / Rollback

- 灰度方式：
  - `budget_warning` 事件为附加事件，前端可忽略，不影响现有 deep task 流程。
  - `harness eval --gate` 为发布前显式门禁，不默认阻塞日常开发命令。
  - 导出链路的超时与重试通过 `deep_task_external_timeout_seconds`、`deep_task_external_retry_limit` 控制，可逐步收紧。
- 回滚方式：
  - 若预算告警噪音过高，可将预算阈值调高，或在前端继续忽略 `budget_warning`。
  - 若导出重试引发异常，可将 `deep_task_external_retry_limit` 调整为 `0`，退回单次尝试。
  - 若发布门禁影响节奏，可暂时移除 `--gate` 调用，仅保留失败摘要输出。
  - `task_id`、`attempt_id` 为附加字段，回滚时保持字段兼容，不需要迁移旧会话。

## Open Questions

- 预算阈值的默认粒度是 token、耗时还是模型调用次数的组合。
- 核心 Recipe 基准集的维护来源应偏真实样本还是合成样本。
- 前端是否需要单独展示预算告警，还是只在日志和回放结果中体现。
