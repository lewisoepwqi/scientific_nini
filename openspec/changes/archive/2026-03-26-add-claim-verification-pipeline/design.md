## Context

前一个 change 已经定义了 `claim_id`、来源记录、Evidence Block 和 METHODS 台账，但这些结构仍然只是“可回溯”，还不是“已验证”。科研输出的核心风险不在于没有引用，而在于系统可能把证据不足或互相冲突的结论包装成确定结论。

因此需要把 Claim 校验单独作为一个 change：它消费前一个 change 产生的证据结构，不重新定义来源模型，而是增加验证状态、评分与摘要准入规则。

## Goals / Non-Goals

**Goals:**
- 对可验证结论执行抽取、证据对齐与状态判定。
- 为结论输出验证状态、置信度评分与原因摘要。
- 将验证状态接入报告摘要与引用展示，形成系统级约束。

**Non-Goals:**
- 不在本 change 中新增来源抓取器或更改来源归一化字段。
- 不在本 change 中新增导出模板或独立证据审查界面。
- 不在本 change 中修改 METHODS 台账记录方式。

## Decisions

### 决策 1：校验流水线以后处理方式接入，而不是阻塞所有生成步骤

Claim 校验主要作用于最终结论，而不是每一步工具执行。因此流水线应在回答或报告草稿生成后作为后处理执行，再影响最终摘要和展示状态。

备选方案：
- 方案 A：每生成一句话就实时同步校验。
- 方案 B：在关键回答或报告草稿完成后批量校验。

选择 B，因为它更容易复用现有产物组装流程，也更可控。

### 决策 2：状态机优先少而稳

MVP 阶段只引入 `verified`、`pending_verification`、`conflicted` 三个核心状态，避免过早设计过细的状态集合。

备选方案：
- 方案 A：增加更多细粒度状态，如 `unsupported`、`partially_supported`、`outdated`。
- 方案 B：先收敛到 3 个核心状态。

选择 B，因为当前目标是先影响摘要准入和用户理解，不是做学术审稿级别的复杂分类。

### 决策 3：摘要准入规则以“默认排除未验证结论”为硬约束

如果只是给未验证结论打标签，但仍允许直接进入最终摘要，可信输出目标无法落地。因此需要把摘要准入写成报告生成层的正式规则。

备选方案：
- 方案 A：只提示风险，不限制摘要内容。
- 方案 B：默认排除未验证与冲突结论，仅在单独区域展示。

选择 B，因为它把可信输出从提示词策略升级为系统行为。

## Risks / Trade-offs

- [校验延迟增加] -> 先在关键输出节点执行批量校验，不对每条中间文本实时触发。
- [评分口径不稳定] -> 先输出评分和原因摘要，保持规则可配置，避免将具体算法硬编码到 spec。
- [过度保守导致摘要过短] -> 对待验证结论单独展示，避免信息完全丢失。
- [前后 change 依赖紧密] -> 明确本 change 依赖 `claim_id` 和来源记录，不重新定义基础数据契约。

## Migration Plan

1. 在后端增加 Claim 抽取与验证流水线，消费已有 `claim_id` 和来源记录。
2. 让报告摘要与引用详情读取验证状态，但默认 UI 仍保持兼容。
3. 分阶段开启“未验证结论不进入摘要”规则，可先在灰度开关下运行。
4. 如校验质量不稳定，可暂时关闭摘要准入限制，仅保留状态输出用于观察。

## Open Questions

- Claim 抽取是否完全基于规则，还是允许模型参与兜底。
- 置信度评分是否需要暴露数值给终端用户，还是仅用于内部和标签映射。
- 冲突检测是否需要区分“来源直接矛盾”和“证据不足导致的不确定”。
