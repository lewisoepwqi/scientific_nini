## Context

当前 Nini 已具备报告资源、图表资源和工作区文件管理，但交付层仍以“单个文件生成后手动下载”为主。对于科研场景，这不足以支撑论文、汇报和项目交付，需要把不同类型产物收束到统一的导出与产物管理管线中。

这个 change 只做产物交付层，不再处理入口、证据绑定或可信校验。它建立的是“如何把已生成的可信内容导出成可交付物”的标准流程。

## Goals / Non-Goals

**Goals:**
- 为 deep task 项目建立统一产物清单与导出作业模型。
- 为报告提供 Word/PPTX/LaTeX 等模板化导出能力。
- 为图表产物补齐标准导出 metadata 与格式状态。
- 在工作区提供基础版本号和打包下载入口。

**Non-Goals:**
- 不实现高级版本 diff、历史回滚或跨会话产物聚合。
- 不在本 change 中新增证据校验或 METHODS 记录逻辑。
- 不在本 change 中处理运行可观测、成本预算或回归门禁。

## Decisions

### 决策 1：把导出视为显式导出作业

如果继续沿用“哪里有文件就直接下载哪里”的模式，后续无法稳定支持多格式、模板和项目级打包。导出必须成为具备输入、状态和输出的显式作业。

备选方案：
- 方案 A：保留分散式下载入口。
- 方案 B：统一为导出作业与项目产物清单。

选择 B，因为它能统一报告、图表和后续更多产物类型。

### 决策 2：项目产物清单只承诺 v0 基础字段

MVP 阶段只记录最小必要字段：类型、资源标识、版本号、来源任务、时间和格式。高级 diff 与回滚后置。

备选方案：
- 方案 A：一次性设计完整版本控制系统。
- 方案 B：先做基础产物清单和版本号。

选择 B，因为当前目标是交付能力，不是完整的资产管理系统。

### 决策 3：报告与图表共用同一产物登记模型

虽然报告和图表的生成路径不同，但项目交付视角下它们都应该是“可导出的正式产物”。因此需要共用同一产物登记模型，只在类型字段上区分。

备选方案：
- 方案 A：报告、图表各自维护独立下载清单。
- 方案 B：共用统一产物登记与下载模型。

选择 B，因为它能减少后续工作区 UI 和下载逻辑分叉。

## Risks / Trade-offs

- [模板体系膨胀] -> 先限制为少量高频模板和格式，模板包设计可扩展但默认收敛。
- [导出作业变慢] -> 导出作业独立建模并记录状态，避免用户误以为卡死。
- [产物索引重复] -> 强制所有正式导出结果走统一登记流程，减少旁路文件。
- [工作区 UI 复杂度增加] -> 只提供项目产物视图和多选下载，不在本阶段加入高级筛选与比对。

## Migration Plan

1. 先定义项目产物清单与导出作业的数据结构。
2. 接入报告导出，再接入图表 metadata 与格式声明。
3. 在工作区增加项目产物视图与打包下载入口。
4. 如导出稳定性不足，可先保留旧下载入口，同时灰度新导出作业。

## Open Questions

- 首批 Word/PPTX/LaTeX 模板应按科研论文、汇报 deck 还是通用研究摘要优先。
- 打包下载是否需要包含原始中间产物，还是仅包含正式导出结果。
- 项目产物版本号是否由资源层自动递增，还是按导出模板维度分组。
