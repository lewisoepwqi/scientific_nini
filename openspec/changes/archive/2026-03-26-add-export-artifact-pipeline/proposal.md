## Why

前面几个 change 已经解决了任务入口、证据绑定和可信校验，但 Nini 的最终交付物仍主要停留在会话内文件和基础下载，离“可直接用于论文/汇报”的标准产物还有明显差距。需要单独的导出与产物管线 change，把报告、图表和项目级产物统一到一个可管理、可模板化、可打包下载的交付层。

## What Changes

- 新增 `artifact-export-pipeline` 能力，定义项目级产物清单、模板化导出与打包下载流程。
- 扩展报告生成能力，使报告会话支持 Word、PPTX 与 LaTeX 等模板化导出。
- 扩展图表渲染能力，使图表产物统一携带标准 metadata，并满足导出清单要求。
- 扩展工作区能力，提供产物列表、基础版本号与打包下载入口。
- 非目标：本 change 不包含高级版本差异对比、历史回滚、跨会话记忆与新一轮证据校验逻辑。

## Capabilities

### New Capabilities

- `artifact-export-pipeline`: 定义项目级产物清单、模板选择、导出作业与打包下载规则。

### Modified Capabilities

- `report-generation`: 扩展报告会话的模板化导出格式与导出作业行为。
- `chart-rendering`: 扩展图表产物 metadata 与标准导出清单。
- `workspace`: 扩展工作区中的产物列表、版本号与打包下载行为。

## Impact

- 后端会影响报告导出、图表导出、工作区产物索引与下载接口。
- 前端会影响工作区产物列表、导出入口与下载状态展示。
- 可能需要引入文档模板资源或模板配置，但优先复用现有导出链路与工作区文件系统。
- 验证方式至少包括 `pytest -q`、导出回归测试，以及 `cd web && npm run build`。
